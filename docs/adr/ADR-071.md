# ADR-071: Telegram-Erlaubnis-Haken fürs Folgenreiche (S4b Scheibe 2, Zuschnitt A)

## Problem / Kontext
Der Wochenend-Bauplan-Leitstern: „Jarvis baut wie Claude — voller (kuratierter)
Werkzeugkasten, **Freigaben & Anweisungen aufs Handy**." Scheibe 1 (kuratierter
Dev-Werkzeugkasten, `agent_dev_tools`) ist gebaut: der Bau-Agent darf Tests/Gate
laufen lassen und git LESEN. Alles Folgenreiche (git push, rm, freie Shell) ist
schlicht **verboten** (Allowlist fail-closed). Der ursprünglich geplante
`--permission-prompt-tool` existiert in der CLI nicht (Befund 11.07.); der
korrigierte Mechanismus ist ein **PreToolUse-Hook** über `--settings`.
PO-Go 13.07.: „Ja mach mit A weiter" (A = kuratiert-weit, Folgenreiches per Haken).

## Entscheidung
Ein PreToolUse-Hook fragt den PO bei folgenreichen Bash-Befehlen **live per
Telegram** (ja/nein); ohne Ja bleibt es beim Nein.

**Bausteine:**
1. `core/hook_gate.py` — geteilte, stdlib-only Basis: `CURATED_BASH` (Single
   Source; `agent_backend._DEV_BASH` ist jetzt ein Alias — keine Drift zwischen
   Allowlist und Hook), `is_curated_bash` (konservativ: Verkettungen/
   Substitutionen nie kuratiert), `HookMailbox` (Datei-Briefkasten in
   `memory_dir/hook_requests`, atomar, KEIN Netzwerk-Endpunkt), 
   `write_hook_settings` (erzeugt die `--settings`-Datei; Mailbox/Timeout als
   **Argumente** im Hook-Kommando, kein Env).
2. `scripts/agent_permission_hook.py` — der Hook-Prozess: kuratiert-sicher →
   KEINE Entscheidung (normale Allowlist-Prüfung); sonst `mailbox.ask()` →
   Ja → `permissionDecision: allow`, Nein/Timeout/JEDER Fehler → `deny`.
   Ausgabe in beiden Schemata (hookSpecificOutput + decision) für CLI-Toleranz.
3. Backend/Delegate: `work(..., hook_settings=...)` hängt `--settings` an —
   NUR im Dev-Modus (wide_tools) und nur wenn konfiguriert; als bedingtes
   kwarg (ältere Backends/Fakes unverändert).
4. Runtime: erzeugt Settings beim Start (Opt-in `agent_permission_hook_enabled`);
   je Delegation lauscht ein Watcher-Thread (1,5 s) auf Anfragen, pusht die
   Frage geschwärzt aufs Handy („🔐 Der Bau-Agent möchte ausführen: «…» —
   Erlauben?") und armiert ein kanalgebundenes ja/nein-Angebot (genau EINE
   offene Frage zugleich). `_process_inner` konsumiert die Antwort VOR allem
   anderen und schreibt sie in die Mailbox.

## Sicherheit (Pflöcke des Bauplans, alle erfüllt)
- **Fail-closed an JEDER Bruchstelle (Pflock 2):** Hook fehlt/stürzt/Timeout,
  Runtime weg, Telegram-Funkloch, Frage verfallen, „ja" nach Ablauf → **NEIN**.
  Die CLI-Allowlist bleibt eng; der Hook kann nur mit PO-Ja ERWEITERN — den
  Haken abzuschalten macht Jarvis also STRENGER, nie laxer.
- **Kern bleibt extra gesperrt (Pflock 1):** Der Haken existiert nur im
  Dev-Modus, und der wirkt nie für Jarvis' eigenes Repo (Selbst-Repo-Wächter).
- **Der Klassifikator fängt das Folgenreiche (Pflock 3):** kuratierte Muster =
  exakt die Allowlist (Single Source); Verkettungen („pytest && rm") sind NIE
  kuratiert → landen beim Haken; leerer Befehl → deny. Getestet.
- Kanalbindung: nur der Frage-Kanal (Telegram) kann antworten; fremde Kanäle
  lassen die Frage armiert.
- Antwortfenster 100 s < Hook-Timeout 110 s < CLI-Hook-Timeout 125 s — die
  saubere Deny-Antwort gewinnt immer vor dem harten CLI-Abbruch.

## Alternativen
- **HTTP-Endpunkt statt Datei-Mailbox:** verworfen — neuer Angriffsweg/Port;
  Dateien sind lokal, atomar, und „Runtime weg = keine Antwort = NEIN" ist
  strukturell fail-closed.
- **Bash breit erlauben + Hook als einzige Bremse:** verworfen — ein
  abgestürzter/fehlender Hook wäre fail-OPEN. So bleibt die Allowlist die
  harte Grenze.
- **Zuschnitt B (voll offene Shell):** vom PO nicht gewählt.

## Falsifizierbarkeit
Unzureichend, wenn der Agent trotz Nein/Timeout folgenreiche Befehle ausführt
(→ CLI-Schema-Drift beim Hook-Output prüfen), oder wenn die Fragen den PO
nerven/zu spät erreichen. Beweis: beaufsichtigter Live-Lauf (ein echter Bau,
der z. B. `git push` versucht → Frage aufs Handy → Nein → Agent arbeitet ohne
weiter). Steht noch aus, zusammen mit dem Scheibe-1-Rauchtest („pytest && rm"
wird abgewiesen).
