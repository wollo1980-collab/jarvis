# ADR-025: Jarvis-Runtime v1 - minimales Gerüst (Queue, Worker-Thread, Konsolen-Dummy-Kanal)

## Problem
ADR-024 legte die Architekturrichtung für eine künftige Jarvis-Runtime fest (koordinierender Einstiegspunkt für mehrere gleichzeitige Kanäle, `queue.Queue`+Worker-Thread statt `asyncio`, Koexistenz mit `main.py`/`telegram_main.py`), aber bewusst ohne Implementierung. Diese ADR dokumentiert die tatsächliche Umsetzung von **Runtime v1** - dem kleinstmöglichen, funktionierenden Architekturbaustein, der ADR-024s Grundgerüst nachweist, ohne bereits einen echten Produktivkanal (UI, Tray, Wake-Word, Telegram) zu enthalten.

## Entscheidung

### 1. Neue Datei `jarvis_runtime.py`, keine bestehende Datei geändert
Top-level, neben `main.py`/`telegram_main.py` (kein neues `runtime/`-Package - YAGNI, gleicher Präzedenzfall wie die flache `commands/`-Struktur, ADR-013). `main.py`, `telegram_main.py`, `core/*`, `commands/*`, `executor/*` bleiben vollständig unverändert - Runtime v1 ist rein additiv und verdrahtet ausschließlich bereits bestehenden Code neu.

### 2. `JarvisRuntime`: Core-Stack einmalig instanziiert
1:1 Wiederverwendung der `main.py`-Verdrahtungsreihenfolge: `AIEngine` → `Planner` → `Executor` (mit einem fail-closed Speech-Adapter, siehe Punkt 4) → `JsonMemoryStore`/`LongTermMemory` → `memory_commands.configure(...)`, `reports_commands.configure(ai)`, `monitor_commands.configure(ai)`. `ai` ist injizierbar (`Optional[AIEngine] = None`, gleiches Testbarkeits-Muster wie `JarvisBridge` in `telegram_main.py`).

### 3. `queue.Queue` + ein Worker-Thread, kein `asyncio`
`JarvisRuntime.submit(text, reply_callback)` legt `(text, reply_callback)` in eine `queue.Queue`. Ein einzelner Worker-Thread (`start()`/`stop()`) verarbeitet Nachrichten **sequenziell** in Eingangsreihenfolge über denselben `Planner`/`Executor` - identisch zum bisherigen Ein-Kanal-Verhalten, nur dass jetzt beliebig viele Kanäle gleichzeitig in die Queue schreiben dürfen. Stop-Pfad: ein Sentinel-Objekt wird in die Queue gelegt, der Worker beendet sich sauber, `stop()` wartet per `thread.join()`.

**Worker stirbt nicht still bei Fehlern (explizite Vorgabe):** Jede Nachrichtenverarbeitung läuft in einem `try/except Exception`-Block innerhalb der Worker-Schleife - eine Ausnahme wird geloggt (`logger.exception`), die Schleife läuft mit der nächsten Nachricht weiter. Ein fehlerhafter `reply_callback` (z. B. Kanal bereits weg) wird separat abgesichert und reißt den Worker ebenfalls nicht mit.

### 4. Fail-closed Speech-Adapter, dupliziert statt importiert
`_RuntimeSpeech` (privat, in `jarvis_runtime.py`) implementiert `say()`/`listen()` fail-closed - gleiches Prinzip wie `TelegramSpeech` in `telegram_main.py` (ADR-018): Sicherheitsstufe-2/3-Commands, die eine Bestätigung über den Executor anfordern würden, bekommen `listen()` = `""` zurück (nie eine erfundene Bestätigung) und werden dadurch abgelehnt. Bewusst **dupliziert statt importiert** - ein echter Import von `telegram_main.py` würde eine unnötige Abhängigkeit von `python-telegram-bot` in die Runtime ziehen, obwohl Telegram in v1 gar nicht Teil der Runtime ist.

### 5. `ConsoleDummyChannel`: einziger Kanal in v1
Liest interaktiv von der Konsole (wie `main.py`s Schleife), reicht jede Zeile über `runtime.submit(text, reply_callback)` weiter, wartet synchron (per `threading.Event`) auf die Antwort und druckt sie. Kein Produktivkanal - beweist nur, dass das Runtime-Gerüst (Core-Stack + Queue + Worker-Thread + sauberer Shutdown) tatsächlich funktioniert, und erlaubt PO denselben Typ manueller Smoke-Test wie bei Telegram Phase 1.

### 6. Bewusst NICHT enthalten
UI, Tray, Wake-Word, Telegram-Integration in die Runtime, Windows-Autostart, abstraktes `Channel`-Interface/Protokoll (erst beim zweiten echten Kanal - YAGNI, gleicher Präzedenzfall wie die `configure()`-Duplizierung zwischen `reports.py`/`monitor.py`), generische Message-/Envelope-Klasse mit Metadaten (ein einfaches Tupel reicht), Persistenz/Recovery der Queue, echte Nebenläufigkeits-Absicherung (Locking) in `JsonMemoryStore`/`Executor` (nicht nötig, da die Queue serialisiert), `asyncio`.

## Begründung
- Serialisierte Verarbeitung über eine einzelne Queue löst das in ADR-024 identifizierte Locking-Problem bei `memory_data/` (kein Locking vorhanden, ADR-018 umgeht es bisher nur durch "ein Kanal zur Zeit"), ohne `JsonMemoryStore`/`Executor` anzufassen - deutlich kleinere, sicherere Änderung als echte Thread-Safety im Core.
- Fail-closed Speech-Adapter statt eines echten Bestätigungsmechanismus: Design Principle 4 (Safety First) - ein Konsolen-Dummy-Kanal ohne festgelegtes UI kann keine verlässliche Bestätigungs-UX bieten; lieber Stufe-2/3-Aktionen sicher blockieren als eine Bestätigung zu erfinden (gleiches Prinzip wie `TelegramSpeech`, ADR-018).
- Ein einziger, minimaler Dummy-Kanal statt sofort UI/Tray/Wake-Word: Regel 6 (keine Architecture Astronautics) - das Runtime-Gerüst selbst ist die architektonische Neuerung, nicht ein bestimmter Kanal; ein echter Kanal lässt sich erst sinnvoll bewerten, wenn das Gerüst nachweislich funktioniert.
- Duplizierter statt importierter Fail-closed-Adapter: vermeidet eine unnötige `python-telegram-bot`-Abhängigkeit in der Runtime für ein Feature, das ohnehin nur ein Sicherheits-Fallback ist (wenige Zeilen Code).
- Kein abstraktes `Channel`-Interface: Regel 6/YAGNI - eine Abstraktion für "mehrere Kanäle" ist verfrüht, solange nur ein Kanal existiert; wird geprüft, sobald ein zweiter echter Kanal ansteht (gleiche Regel wie beim `configure()`-Muster, ADR-020).

## Alternativen erwogen
- **`asyncio` statt Thread+Queue:** verworfen (ADR-024, bestätigt) - mehr Komplexität als für v1 nötig.
- **`Channel`-Basisklasse/Protokoll bereits jetzt einführen:** verworfen - verfrühte Abstraktion für einen einzigen Kanal.
- **`TelegramSpeech` aus `telegram_main.py` importieren statt `_RuntimeSpeech` zu duplizieren:** verworfen - zöge `python-telegram-bot` als Abhängigkeit in die Runtime, obwohl Telegram nicht Teil von v1 ist.
- **Echte Bestätigungs-UX im `ConsoleDummyChannel` (Ja/Nein über die Konsole):** verworfen - der Dummy-Kanal soll bewusst minimal bleiben; Stufe-2/3-Unterstützung ist ein Feature für einen echten Kanal, nicht für das Beweis-Gerüst.
- **Queue-Persistenz (z. B. auf Platte) für Absturzsicherheit:** verworfen - unnötige Komplexität für v1, akzeptables Risiko (Nachrichten gehen bei Absturz verloren, kein Multi-Request-Broadcasting an einen einzelnen Nutzer).

## Konsequenzen
- Neue Datei `jarvis_runtime.py`, neue Testdatei `tests/test_jarvis_runtime.py`.
- **Keine** Änderung an `main.py`, `telegram_main.py`, `core/*`, `commands/*`, `executor/*` (per `git diff --stat` verifiziert).
- Jarvis-Eigenstart-Implementierung bleibt weiterhin verschoben - ob Runtime v1 dafür bereits ausreicht oder eine spätere Runtime-Version nötig ist, bleibt eine eigene, spätere Entscheidung.
- Ein zweiter echter Kanal (UI, Tray, Wake-Word, Telegram) ist der natürliche Anlass, das `Channel`-Interface tatsächlich einzuführen.
- Kein gleichzeitiger Betrieb mit `main.py`/`telegram_main.py` gegen dieselbe `memory_data/` bedacht (wie ADR-018) - bleibt ein bekanntes, nicht gelöstes Risiko für v1.
