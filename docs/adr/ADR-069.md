# ADR-069: build_project greift ein bereits von Jarvis angelegtes Gerüst wieder auf

## Problem / Kontext
Der **erste echte Live-Lauf der Königsdisziplin** (PO 12.07.2026) blieb stecken.
Jarvis schlug proaktiv „«Erinnerungs-Manager» — Sag «Bau mir Erinnerungs-Manager»"
vor (Stufe 1 wirkte). Die Zustimmung („Ja, klingt gut. Mach das") routete aber auf
**`start_project`** (nur Gerüst) statt auf **`build_project`** (der Einschuss:
Gerüst + Schreib-Freigabe + Bau). Ergebnis:
- `start_project` legte `C:\KI\erinnerungs-manager` an, erteilte aber **keine**
  Schreib-Freigabe und verwies auf ein „Onboarding-Interview" → `project_continue`.
- `project_continue` ist **fail-closed**: nur Projekte in `agent_write_repos`
  (= `jkc`) sind schreibbar; ein frisch vorgeschlagenes Projekt steht dort nie →
  „nicht freigegeben (freigegeben: jkc)".
- Sobald der Ordner existierte, wäre auch ein korrektes „Bau mir X" → `build_project`
  an `scaffold_project`s „Verzeichnis existiert bereits" gescheitert. **Sackgasse.**

PO-Grundsatz dazu: *„Wenn Jarvis bauen darf und kann, dann sollte er auch in dem
Ordner bauen dürfen, den er angelegt hat. Er soll generell dasselbe dürfen wie
Claude — mit meiner Freigabe. Nur sein eigenes Repo/sein Code ist Tabu."*

## Entscheidung
`build_project` wird **wiedereinstiegsfähig**: existiert `projects_root/<name>`
bereits **und** ist es ein echtes, von Jarvis nach dem AI Project Framework
angelegtes Gerüst, wird es **aufgegriffen** (Freigabe + Bau) statt abzubrechen.
Andernfalls (Ordner existiert, aber kein Jarvis-Gerüst) bleibt es fail-closed.

Die Erkennung `core/agent_grants.is_framework_scaffold(path)` verlangt zwei
Merkmale: `docs/PROJECT_STATE.md` existiert UND `docs/logbook.md` trägt die
PROJECT_INIT-Pflichtzeile „Abgeleitet aus AI Project Framework". Diese Zeile
schreibt nur `scaffold_project` in Kindprojekte — Jarvis' eigenes Repo und der
öffentliche Export tragen sie **nicht**.

## Sicherheit (der Käfig wird NICHT aufgeweitet)
- `may_grant` (die vier ADR-059-Wächter) bleibt **unverändert** und wird beim
  Wiedereinstieg **erneut** geprüft: nur unter `projects_root`, nie die Wurzel,
  **nie Jarvis' eigenes Repo** (`BASE_DIR`) oder darunter.
- Der **Gerüst-Marker** ist die zusätzliche, entscheidende Schranke: `projects_root`
  = `C:\KI` enthält auch `jarvis-public-export` (Geschwister, nicht unter BASE_DIR);
  `may_grant` allein würde es erlauben — `is_framework_scaffold` lehnt es ab.
  Am realen Baum verifiziert: erinnerungs-manager → aufgreifbar; jarvis-public-export
  → abgelehnt; jarvis (Selbst) → abgelehnt.
- Unverändert: Bestätigung (requires_confirmation), Sandbox, Selbstprüfung, Auto-
  Commit-Ampel, sitzungs-lokale (nicht persistierte) Freigabe. Selbst-Modifikation
  bleibt strukturell gesperrt.
- **Nicht** Teil dieses ADR (bewusst, eigener Schritt): volle Werkzeug-Parität mit
  Claude (beliebige Shell/Netz/out-of-scope-Schreibzugriff) = S4b „Käfig aufweiten".

## Alternativen
- **`project_continue` frische Projekte freigeben lassen:** weiter reichende
  Käfig-Änderung (der zweistufige Pfad ist für große Framework-Projekte gedacht);
  verworfen zugunsten des schmaleren Einschuss-Wiedereinstiegs.
- **Nur das Routing fixen (Zustimmung → build_project):** Routing ist nicht-
  deterministisch (ADR-068); der Wiedereinstieg macht ein Fehl-Routing selbst-
  heilend statt tödlich und ist damit robuster.
