# ADR-034: Agenten-Delegation v1 – read-only Repo-Analyse (erste Fähigkeit + Backend)

## Problem/Kontext

ADR-033 hat den **modellunabhängigen Delegationsprozess** festgelegt, bewusst
ohne konkreten Agenten und ohne Fähigkeit. Dieser ADR ist der angekündigte
**Aufsatz**: Er wählt das erste Agenten-Backend und definiert die erste,
kleinste, sichere Fähigkeit — die read-only Repo-Analyse.

**Machbarkeits-Check (2026-07-06, PO-freigegeben, read-only):**
- `claude -p` (Claude Code CLI, Headless-/Print-Modus) läuft als Subprozess und
  liefert echte Antworten.
- **Read-only ist nachweisbar:** Git-Status vor und nach dem Lauf leer; nur
  `Read`/`Grep`/`Glob` erlaubt (im Print-Modus sind Schreib-/Ausführ-Tools nicht
  verhandelbar).
- **Auth trägt über den Account-Login** — das anfängliche „Not logged in" lag nur
  am unfertigen Onboarding. Kein zwingender bezahlter API-Key für den Grundlauf.

## Entscheidung

### 1. Erstes Backend: Claude Code CLI
Installiert und angemeldet, ausgereifter `-p`-Headless-Modus mit erzwingbarem
Read-only (`--allowedTools Read Grep Glob`), stark in Codebase-Analyse.
Modellneutral (ADR-033): ein austauschbares Backend, kein Lock-in — Codex/GPT
folgen später hinter demselben Kontrakt.

### 2. Erste Fähigkeit: read-only Repo-Analyse
Neuer Command (Arbeitsname `delegate_analysis`): „analysiere `<repo>`: `<Frage>`"
→ Jarvis startet Claude Code headless read-only im allowlisteten Repo → liefert
die Analyse als reviewbares Ergebnis. **Sicherheitsstufe 0** (keine System-/
Repo-Änderung), konsistent mit `search_web`/`check_mail`. Read-only auf
Agenten-Ebene erzwungen; **keinerlei git-Operation** (kein Branch/Commit/Push).

### 3. Repo-Allowlist
Config-getrieben (analog `mail_accounts`). Start: **genau ein Eintrag**,
`C:\KI\jarvis`. Nicht-allowlistete Pfade → fail-closed abgelehnt.

### 4. Kanal und asynchrone Übergabe
Auslösbar **lokal** (Konsole) und über **Telegram** (neuer read-only Intent in
`ALLOWED_INTENTS`, wie `search_web`/`check_mail`). Eine Agenten-Analyse dauert
Minuten und darf den seriellen Runtime-Worker nicht blockieren → **asynchron**:
sofortige Quittung („mache ich, melde mich") → Hintergrundlauf → **Push des
Ergebnisses** bei Fertigstellung. Ergebnis = Kurz-Zusammenfassung über den Kanal
+ vollständiges Artefakt (`<memory_dir>/delegations/<zeitstempel>.md`) zum Review.

### 5. Guardrails
Read-only erzwungen · keine git-Ops · Allowlist · harte Limits (Timeout, Turn-/
Kostendeckel, Kill-Switch) · vollständiges Logging (Repo · Frage · Backend ·
Dauer · Status ✓/✗/? · Kosten soweit verfügbar) · Trust Boundary (Ergebnis rein
informativ, löst nie eine Aktion aus) · Datenschutz-Hinweis (die Analyse kann
Code-Ausschnitte über Telegram transportieren — eigenes Repo, eigene Chat-ID,
vertretbar; Secrets liegen in Env, nicht im Repo).

### 6. Auth-Strategie
Start auf dem vorhandenen **Account-Login** (null Zusatzkosten/Setup). Ein
dedizierter `ANTHROPIC_API_KEY` (per Env, ADR-018) ist die dokumentierte
**Robustheits-Aufwertung** für den späteren *unbeaufsichtigten* Remote-Betrieb
(OAuth-Token laufen ab → stiller Fehlschlag). Vor Produktivbetrieb ist zu
verifizieren, dass `claude -p` auch aus Jarvis' `pythonw`-Prozess angemeldet ist.

### 7. Agenten-Backend-Kontrakt (Skizze; Code folgt bei der Umsetzung)
Ein `AgentBackend`-Protokoll analog `LLMProvider`:
`analyze(repo, frage, limits) -> Ergebnis(text, status, artefakte, kosten)`, mit
erzwingbarem Read-only-Modus. Claude Code ist die erste Implementierung
(Subprozess `claude -p --allowedTools Read Grep Glob`).

## Begründung
Kleinste sichere Scheibe des Agenten-Arms: read-only = null Schreibrisiko; async
passt zu „während ich unterwegs bin"; modellneutral gemäß ADR-033. Erster echter
Praxistest des Delegationsprozesses → liefert dessen Falsifizierbarkeits-Evidenz.

## Alternativen erwogen
- **Codex zuerst:** verworfen — nicht im PATH, eigene Auth ungeprüft.
- **Sofort Scheibe 2 (Fix auf Branch):** verworfen — unbeaufsichtigte Schreib-/
  Ausführ-Vollmacht ist zu groß für den ersten Schritt.
- **Nur lokal (kein Telegram):** verworfen — der Wert ist „unterwegs"; read-only
  ist remote sicher genug (wie `search_web`/`check_mail`).
- **API-Key-Zwang ab Tag 1:** verworfen — der Account-Login trägt; kein
  Kaufzwang für die erste Scheibe.

## Konsequenzen
- Neuer Command + Core-Baustein (`AgentBackend` + Claude-Code-Implementierung),
  **erste asynchrone Hintergrund-Ausführung** im Projekt, neuer Telegram-Intent,
  Config-Allowlist.
- Die **Umsetzung ist ein eigenes, getrennt freizugebendes Arbeitspaket** —
  dieser ADR ist die Entscheidung, nicht die Implementierung.

## Risiken
- **Token-Ablauf (Auth):** stiller Fehlschlag bei unbeaufsichtigtem Lauf →
  klare Fehlermeldung, später API-Key.
- **Kosten/Quota:** Limits + Kostendeckel.
- **Async-Komplexität:** erster langlaufender Hintergrund-Task → sorgfältig testen.
- **Prompt-Injection über Repo-Inhalte:** read-only + rein informatives Ergebnis
  begrenzen die Fläche (kein Aktions-Trigger).

## Falsifizierbarkeit
Zu revidieren, wenn die erste echte Delegation zeigt, dass read-only Analyse
keinen realen Alltagswert bringt, oder wenn Async/Auth/Kosten die Scheibe
unpraktikabel machen.

## Bewusst NICHT in diesem Schritt
Scheibe 2 (Schreiben/Branch/Fix) · Multi-Agenten · Codex/GPT-Backends ·
Geräte-Ebene · Dienst-Orchestrierung (Kalender) · dedizierter API-Key (nur
vorgemerkt) · jegliche Implementierung/Code.
