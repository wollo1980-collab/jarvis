# ADR-056: Durchsichtiger Agenten-Harnisch (Orchestrier-Geruest)

## Status

**Accepted — PO-Freigabe am 2026-07-11** (🟡, neue Architektur + Sicherheits-/
Lesen-Handeln-Grenze; Delegations-Matrix CONTRIBUTING §3). Umsetzung in
Scheiben. Nach unabhaengigem Review (2026-07-11) um drei Praezisierungen
zur Gruen-Klassifikation geschaerft (Scope-Bindung, selbst-erstellt-vs-fremd,
Auftrag=Arbeitspaket) - vor Scheibe 4 verbindlich (siehe Punkt 4).
Fortschritt: Scheibe 1 (Durchsicht) + Scheibe 2 (Stopp) + Scheibe 3
(interaktives Umlenken) umgesetzt. Offen: Scheibe 4 (Ampel-Gating +
Auto-Commit, Sicherheitskern).

## Problem/Kontext

PO-Zielbild (11.07.2026): Jarvis als **Dirigent**, der mehrere KIs/Tools
(Claude, GPT, spaeter Video/Ton) orchestriert, Schritte selbst erkennt, mit
den Werkzeugen abarbeitet und einen Entwurf vorlegt - der Mensch sagt am
Ende nur „ja, das posten wir". Zwischenziel: Jarvis soll **an eigenen
Projekten so bauen koennen wie Claude in einer Dev-Sitzung** - direkt, mit
mehr Werkzeugen, aber sicher.

Heute laeuft der Agent als **versiegelte Kiste**: `claude -p
--output-format json` (Stapel, ein JSON am Ende), Werkzeug-Kaefig ueber
`--allowedTools`, Ergebnis als Diff zur Sichtung. Es fehlt, was eine
Dev-Sitzung sicher UND direkt macht: Sicht auf jeden Schritt, ein Stopp,
kluges Nachfragen - und die Faehigkeit, andere Tools anzustecken.

Kern-Einsicht (PO-Dialog): Sicherheit kommt NICHT aus „mehr Regeln" (der
Agent IST Claude, hat dieselben), sondern aus **Sicht + Stopp +
begrenztem Radius** und dem Menschen im Loop fuer das Folgenreiche.

## Entscheidung

Ein **generischer Agenten-Harnisch** um den bestehenden, werkzeug-agnostischen
`AgentBackend`-Vertrag (ADR-036) - fuenf Eigenschaften, in Scheiben baubar:

1. **Durchsicht (Streaming):** Der Lauf emittiert pro Schritt ein Ereignis
   (Werkzeug-Aufruf, Datei-Aenderung, Gedanke), das live ins UI wandert
   (bestehende Timeline/SSE). Bei Claude Code ueber `--output-format
   stream-json`. Der Ereignis-Kanal ist **generisch** - jeder Backend-Adapter
   liefert dieselbe Ereignis-Form.
2. **Stopp:** Ein sichtbarer Abbruch mitten im Lauf. Der Kill-Switch existiert
   (`cancel_event` + harter Prozess-Kill, ADR-035) - nur an einen UI-Knopf
   haengen.
3. **Interaktiv (zweiseitig):** Der Kanal ist von Anfang an bidirektional
   angelegt, damit der Nutzer den Agenten SPAETER waehrend des Laufs
   umlenken kann („mach's anders"). Erste Fassung: zusehen + stoppen +
   Gates; fluessiges Mitreden folgt, ohne den Kanal umzubauen.
4. **Klug nachfragen = die Ampel als Politik.** Klassifikation = die
   Delegations-Matrix (CONTRIBUTING §3), WOERTLICH. Drei Praezisierungen
   (Review 2026-07-11, vor S4 verbindlich):

   a. **Gruen ist an den SCOPE gebunden, nicht an Reversibilitaet.** Die
      Matrix sagt „umkehrbare Arbeit IM FREIGEGEBENEN SCOPE". Reversibel ist
      NOTWENDIG, nicht hinreichend. Der Harnisch prueft also „ist das *im
      Auftrag*?", nicht nur „ist das umkehrbar?". Eine reversible Aenderung
      ausserhalb des Auftrags-Scopes ist 🟡 (Scope-Aenderung) → Rueckfrage,
      nie Freibrief.
   b. **Selbst-erstellt vs. fremd.** Die Matrix nennt „Loeschen/Ueberschreiben
      *nicht selbst erstellter* Dateien" ausdruecklich 🟡. Der Harnisch
      unterscheidet also selbst-angelegt vs. vorgefunden - nicht nur
      „im Repo vs. ausserhalb".
   c. **Der Auftrag IST die PO-Freigabe des Arbeitspakets.** Dein Bau-Auftrag
      an Jarvis deckt genau *dieses* Arbeitspaket ab. Der Auto-Commit betrifft
      nur das verifizierte, im-Scope-liegende Ergebnis DIESES Auftrags - kein
      Freibrief fuer kuenftige, unbesprochene Aenderungen. So bleibt die
      Sicherheitsstufen-Tabelle (Handbook) und die Praxis deckungsgleich.

   Damit: Triviales IM AUFTRAG (lesen / selbst-erstellte Repo-Dateien
   editieren / Code+Tests ausfuehren / Web lesen) → Freibrief, kein
   Nachfragen. Folgenreiches ODER ausserhalb des Scopes ODER fremde Dateien
   (loeschen ausserhalb, Netz-Schreiben, zahlen, force-push, Scope-Kriechen,
   ROTE Kategorie) → Erlaubnis-Haken (`--permission-prompt-tool`) fragt im
   UI. GRUENE Kategorie im Scope + Selbstpruefung gruen (ADR-055) → Jarvis
   committet selbst; ROTE → Vorlage + Freigabe. Freigabe-Protokollierung
   erbt mit (Commit-Message referenziert die Freigabe).
5. **Werkzeug-agnostisch (Steckdose):** Harnisch + Politik haengen am
   `AgentBackend`-Vertrag, nicht an Claude Code. Ein neues Tool (Codex, ein
   Bild-/Video-Dienst) = ein Adapter hinter dem Vertrag und erbt Sicht,
   Stopp, Erlaubnis. Das ist die Voraussetzung fuer die spaetere
   Multi-Tool-Orchestrierung.

## Zwei Schichten (bewusst getrennt)

- **Harnisch (diese ADR):** Sicherheit + Kontrolle. Macht „ich sag nur noch
  ja" ueberhaupt sicher.
- **Orchestrierung (spaeter, eigene ADR):** Ziel zerlegen → Werkzeug waehlen
  → Ergebnisse weiterreichen → Entwurf bauen. Waechst Werkzeug fuer Werkzeug.

## Sicherheitsmodell

Wie ADR-055: bounded blast radius + Mensch-im-Loop. Selbst-Modifikation von
Jarvis' eigenem Kern/Neustart bleibt EXTRA gesperrt (nie den Ast absaegen,
auf dem er sitzt) - eigene Projekte sind unkritisch. Auto-Commit ist eine
der SICHEREREN Befugnisse (lokal, umkehrbar, privates Remote); das Heikle
ist die gruen/rot-Klassifikation → Sicht + Stopp + Selbstpruefung sind das
Netz.

## Scheiben (Reihenfolge)

1. **Durchsicht:** stream-json → Ereignisse → UI-Timeline. Gibt KEINE neue
   Macht, reine Beobachtung. Fundament.
2. **Stopp-Knopf** → cancel_event.
3. **Interaktives Umlenken** (zweiter Kanal-Zweig scharf).
4. **Ampel-Gating + Auto-Commit** (Erlaubnis-Haken, Klassifikation, Commit).
5. Danach: erster fremder Backend-Adapter (Steckdose beweisen), dann
   Orchestrierung.

## Alternativen

- **Alles im Stapel lassen (Status quo):** verworfen - kein „wie Claude".
- **Nur Regeln/Prompt verschaerfen:** verworfen - Regeln sind Wahrscheinlich-
  keit, kein Netz (der PO-Dialog hat's herausgearbeitet).
- **Claude-Code-spezifischer Harnisch:** verworfen - jedes neue Tool waere
  ein Umbau; generisch am Vertrag macht neue Tools zu kleinen Steckern.

## Bewusst NICHT (in dieser ADR)

- Keine Selbst-Modifikation von Jarvis' Kern/Neustart ohne extra Sperre.
- Keine Multi-Tool-Orchestrierung (eigene, spaetere ADR).
- Kein Anschluss konkreter externer Dienste (Video/Ton) - je eigener Adapter.

## Falsifizierbarkeit

Gilt als falsch geschnitten, wenn sich zeigt, dass der generische Ereignis-/
Erlaubnis-Kanal die Eigenheiten realer zweiter Backends nicht traegt - dann
Review dieser ADR vor dem zweiten Adapter, nie stillschweigend
Claude-spezifisch verdrahten.
