# ADR-049: Projektstart auf Zuruf — deterministisches Gerüst nach PROJECT_INIT

## Problem/Kontext

Projektentwickler-Kampagne (PO-Auftrag 10.07.2026: „Jarvis soll selbstständig
Projekte entwickeln", erstes Zielobjekt direkt das JKC-Gerüst). Stufe 1 des
Plans: „starte Projekt X" erzeugt das Gerüst eines neuen Zielprojekts. Die
normative Vorlage existiert bereits — `docs/PROJECT_INIT.md` des AI Project
Frameworks verlangt: eigenes Git-Repo außerhalb des Framework-Repos, früher
erster Commit, logbook-Pflichtzeile mit Framework-Commit-Hash (+
charter_version), ADR-Template wörtlich übernommen. Die *Inhalte* (Zweck,
Vision, MVP, Stack) entstehen laut Framework erst im Onboarding-Interview
mit dem PO — vor dem ersten Code.

Zugleich ist dies der erste Command, der außerhalb von `memory_data/` ins
Dateisystem schreibt — eine neue Schreibfläche, die klare Grenzen braucht.

## Entscheidung

1. **`start_project`** (commands/project.py) + pure Gerüst-Logik in
   `core/project_scaffold.py` — **deterministische Schablone, kein LLM**.
   Erzeugt: README, docs/PROJECT_STATE (maschinenlesbarer Kopf),
   CHANGELOG/logbook/framework_feedback, ADR-TEMPLATE (wörtliche Kopie aus
   dem Framework), .gitignore, pytest.ini, Rauchtest; dann `git init -b
   main` + erster Commit (Autor „Jarvis <jarvis@local>" — ehrlich
   nachvollziehbar, wer das Gerüst stanzte).
2. **Ehrlichkeit vor Vollständigkeit:** Die Dokumente sagen ausdrücklich
   „Onboarding ausstehend" — Jarvis erfindet weder Zweck noch Stack. Die
   logbook-Pflichtzeile dokumentiert Herkunft: „Abgeleitet aus AI Project
   Framework Commit `<hash>` am `<Datum>`, charter_version `<v>`."
3. **Schreibgrenzen (fail-closed):** Geschrieben wird nur UNTERHALB der
   Config-Wurzel `projects_root`, nie in existierende Verzeichnisse, nie
   mit Remote/Push. Projektname strikt als Slug (`[a-z][a-z0-9-]{1,39}`,
   keine Pfad-Tricks). `projects_root`/`framework_repo` sind per Default
   leer = Fähigkeit aus. Das Framework-Repo wird ausschließlich GELESEN
   (rev-parse, CONTRIBUTING-Frontmatter, ADR-Template) — READ-ONLY-Regel
   des PROJECT_INIT.
4. **Sicherheitsstufe 2** (`requires_confirmation`): Rückfrage vor jedem
   Projektstart — wie install_program. Bewusst NICHT in der
   Telegram-Standalone-Whitelist.

## Begründung

- Die Framework-Vorgabe existiert und ist erprobt (Prompt Manager) — Jarvis
  setzt sie mechanisch um, statt eine eigene Projektform zu erfinden.
- Deterministisch statt LLM: Ein Gerüst ist eine Schablone; Kreativität
  gehört ins Onboarding (PO-Gespräch), nicht in Verzeichnisnamen.
- Die neue Schreibfläche ist eng: eine Wurzel, nur neue Verzeichnisse,
  Slug-Namen — kleiner als jede Alternative (freier Pfad, Overwrite).

## Alternativen

- **LLM-generiertes Gerüst:** verworfen — nicht reproduzierbar, erfindet
  Inhalte, die dem PO-Interview vorgreifen (Framework-Verstoß).
- **Onboarding-Interview gleich mit in Jarvis (Dialog-Strecke):** später
  denkbar (Stufe-2-Zyklus nutzt das ConfirmationGate); für Stufe 1
  überdimensioniert.
- **Gerüst per Hand/Session statt Command:** verworfen — die Kampagne will
  genau diese Fähigkeit IN Jarvis („er soll mir Produkte entwickeln").

## Konsequenzen

- „Starte Projekt jkc" erzeugt nach Bestätigung das JKC-Repo unter
  `projects_root` — bereit fürs Onboarding, ohne eine Zeile Produktcode.
- Stufe 0 der Kampagne (schreibende Agent-Delegation im Käfig) bekommt
  damit ihr reales Zielrepo; eigenes ADR, eigene Freigabe.
- Zwei neue Config-Felder (`projects_root`, `framework_repo`).

## Falsifizierbarkeit

Widerlegt, wenn das erzeugte Gerüst beim JKC-Onboarding substanziell
umgebaut werden muss (dann war die Schablone falsch — Rückfluss-Kandidat
ins Framework) oder wenn die Slug-/Wurzel-Grenzen legitime Projektstarts
verhindern.

## Bewusst NICHT

- Kein Remote/`gh repo create`/Push — Publikation bleibt PO-Schritt.
- Kein Überschreiben/„Reparieren" existierender Verzeichnisse.
- Kein venv-/Dependency-Setup (projekt-/stackabhängig, folgt nach dem
  Stack-ADR des Zielprojekts).
- Kein Adoptionsmodus für Bestandsprojekte (PROJECT_INIT Teil 1) — eigene
  Entscheidung, falls je gebraucht.

## Status

Accepted (PO-Freigabe 10.07.2026, „Go").
