# ADR-018: Telegram-Fernzugriff (v0.6 Phase 1) - eingeschränkte Whitelist, separater Einstiegspunkt

## Problem
Handbook Kap. 13/16/27 nennen für v0.6 "Handy" nur "Telegram-Bot, Fernzugriff" (Kerninhalt) bzw. Telegram-Bot als empfohlene Einstiegsoption (Kap. 16) - ohne Aussage zu Befehlsumfang, Sicherheitsstufen-Handhabung bei Fernzugriff, Technik (Polling/Webhook) oder Architektur (neuer Prozess vs. main.py-Erweiterung). Fernzugriff über einen öffentlichen Messenger ist außerdem eine grundsätzlich neue Risikoklasse gegenüber der bisher ausschließlich lokalen Konsoleneingabe - das bestehende Sicherheitsmodell (Kap. 10) adressiert nur den Ausführungsschritt selbst, nicht die Frage "wer darf überhaupt mit Jarvis sprechen".

Eine Handbook-Prüfung (Scope/DoD/Architektur/Sicherheitsmodell/Bibliotheken/Registry-Integration/Tests/Risiken) hat diese Lücken aufgedeckt. Wolfgang hat daraufhin folgende Product-Owner-Entscheidungen für Phase 1 getroffen.

## Entscheidung
Neuer, separater Einstiegspunkt `telegram_main.py` (Projekt-Root, parallel zu `main.py`; `main.py` bleibt für die Konsole komplett unverändert):

1. **Befehlsumfang (Whitelist):** Nur `chat`, `remember_fact`, `forget_fact`, `system_status` sind über Telegram erreichbar. Keine Excel-/Report-/KPI-Dateizugriffe, kein `install_program`, kein `shutdown_pc` in Phase 1.
2. **Sicherheitsstufen:** Stufe 0 und die genannten Stufe-1-Speicher-Interaktionen sind remote erlaubt; Stufe 2/3/4 sind remote gesperrt. Umgesetzt über **zwei unabhängige Mechanismen** in `telegram_main.py::rejection_reason()`: (a) die feste `ALLOWED_INTENTS`-Liste, (b) ein zusätzlicher Check auf `command.requires_confirmation` - Letzterer greift auch dann, falls die Whitelist später versehentlich um einen Stufe-2/3-Intent erweitert würde (Defense in Depth).
3. **Mehrschritt-Pläne:** Enthält ein Plan (Planner-Split an "und"/"und dann"/"danach"/";") auch nur einen nicht erlaubten Schritt, wird der **gesamte Plan** abgelehnt (`telegram_main.py::filter_plan()`) - keine Teilausführung. Begründung: über einen Textkanal ohne Rückfragemöglichkeit ist eine klare Ablehnung sicherer als eine teilweise, potenziell verwirrende Ausführung.
4. **Technik:** `python-telegram-bot` (≥ 20), Long-Polling (`Application.run_polling()`). Kein Webhook, kein FastAPI, kein ngrok - kein öffentlich erreichbarer Server nötig, passend für einen privaten Start.
5. **Architektur:** `telegram_main.py` verdrahtet dieselben Bausteine wie `main.py` (`Config`, `AIEngine`, `Planner`, `Executor`, `JsonMemoryStore`, `LongTermMemory`, `commands.memory.configure()`) - Telegram ist ein neuer Ein-/Ausgabekanal (Kap. 7: Voice-Ebene), keine neue Command-Kategorie. Ein neuer `TelegramSpeech`-Adapter erfüllt dieselbe `say()`/`listen()`-Schnittstelle wie `SpeechEngine`, damit `Executor` unverändert wiederverwendet werden kann. Da in Phase 1 nur bestätigungsfreie Intents den Executor erreichen, sollten beide Methoden praktisch nie aufgerufen werden - beide sind bewusst **fail closed** implementiert (loggen einen Fehler, `listen()` liefert `""` statt einer stillschweigenden Bestätigung).
6. **Autorisierung:** Genau eine erlaubte Chat-ID (`JARVIS_TELEGRAM_ALLOWED_CHAT_ID`) und der Bot-Token (`JARVIS_TELEGRAM_BOT_TOKEN`) ausschließlich über Umgebungsvariablen - niemals in `config.json` oder Git (Konsistenz mit `OPENAI_API_KEY`/`ELEVENLABS_API_KEY`). Fehlen sie, bricht `telegram_main.py` beim Start mit einer klaren Fehlermeldung ab (Kap. 4: nie stillschweigend scheitern). Nachrichten von anderen Chat-IDs werden ignoriert, ohne Rückmeldung nach außen.
7. **Kein gleichzeitiger Betrieb** von Konsole und Telegram in Phase 1 (Wolfgangs Entscheidung) - beide teilen sich dieselben `memory_data/`-Dateien, aber es läuft jeweils nur ein Kanal, wodurch das sonst mögliche Race-Condition-Risiko beim gleichzeitigen Schreiben entfällt.
8. **`python-telegram-bot`** ist eine **optionale** Abhängigkeit (auskommentiert in `requirements.txt`, wie die TTS-Backends) - `main.py`/die Konsole benötigen sie nicht, nur wer `telegram_main.py` tatsächlich startet.

**Bewusst KEINE Änderung an** `core/ai.py`, `core/planner.py`, `core/tool_manager.py`, `executor/executor.py`, `main.py` oder `commands/*.py` - alle Beschränkungen leben ausschließlich in `telegram_main.py`.

## Begründung
- Die Trennung von Whitelist und Stufe-Check gibt zwei unabhängige Sicherheitsschichten statt einer einzigen Liste, die bei künftiger Erweiterung leicht vergessen werden könnte.
- Ganzen Plan ablehnen statt Teilausführung folgt Design Principle 4 (Safety First) - eine klare, eindeutige Rückmeldung ist einem teilweise ausgeführten, für den Nutzer schwer nachvollziehbaren Ergebnis vorzuziehen.
- Long-Polling vermeidet die größte zusätzliche Angriffsfläche eines Webhook-Ansatzes (kein öffentlich erreichbarer Server) und passt zu "privater Start" (Wolfgangs Begründung).
- Ein separater Einstiegspunkt statt einer Erweiterung von `main.py` hält die bestehende, bereits getestete Konsolen-Pipeline unangetastet - Fernzugriff ist ein optionaler, klar abgegrenzter Zusatzkanal.
- `TelegramSpeech` fail-closed statt fail-open: sollte die Whitelist-Logik jemals einen Fehler haben und ein bestätigungspflichtiger Befehl doch den Executor erreichen, würde eine stillschweigende "Bestätigung" (fail-open) den gesamten Zweck der Beschränkung unterlaufen - `""` (nicht bestätigt) ist die sichere Grundannahme.

## Alternativen erwogen
- **Webhook + FastAPI + ngrok:** verworfen (Wolfgangs Entscheidung) - würde einen öffentlich erreichbaren Endpunkt erfordern, mehr Angriffsfläche und mehr Infrastruktur für einen privaten Erststart.
- **Vollständiger Befehlsumfang von Anfang an:** verworfen - Fernzugriff ist eine neue Risikoklasse (Wolfgangs Begründung); Stufe-2/3-Aktionen und Datei-Commands bleiben für spätere Phasen vorbehalten.
- **Teilausführung bei gemischten Plänen:** verworfen - siehe Begründung oben.
- **Whitelist-Check im Executor statt in `telegram_main.py`:** verworfen - hätte einen Sonderfall im Executor bedeutet (verboten laut Wolfgangs Vorgabe: keine Sonderfälle in `ai.py`/`Planner`/`Executor`/`ToolManager`, wenn vermeidbar).

## Konsequenzen
- Neuer, vollständig additiver Einstiegspunkt - keine bestehende Datei außer Dokumentation/`requirements.txt` geändert.
- Künftige Phase-2-Erweiterungen (mehr Intents, evtl. Stufe-2-Aktionen mit Telegram-Bestätigung) müssten `ALLOWED_INTENTS` erweitern und die `TelegramSpeech`-Fail-Closed-Haltung explizit überdenken (echte `listen()`-Implementierung wäre dann nötig, aktuell bewusst nicht gebaut).
- `docs/PROJECT_STATE.md` verweist ab jetzt auf `telegram_main.py` als zusätzlichen, optionalen Einstiegspunkt.
