# ADR-007: SYSTEM_PROMPT dynamisch aus der Command-Registry bauen

## Kontext
GPT-Review (01.07.2026, Bewertung 8.5-9/10) bemängelte, dass der
SYSTEM_PROMPT in core/ai.py eine hart codierte "Bekannte Intents"-
Liste enthielt, die zwei echte Probleme hatte:

1. Sie nannte Intents wie `search_google` und `weather`, für die es
   gar keine Commands gibt - die KI konnte also Fähigkeiten
   "erkennen", die sofort mit "kenne ich nicht" scheitern mussten.
2. README.md verspricht: neue Commands hinzufügen, ohne `ai.py`
   anzufassen. Das stimmte nicht - ein neuer Intent musste bisher
   zusätzlich manuell in die Prompt-Liste eingetragen werden, sonst
   hätte die KI ihn nie als möglichen Intent erkannt.

## Entscheidung
`core/ai.py` baut den SYSTEM_PROMPT jetzt zur Laufzeit aus
`commands.REGISTRY`:

- `_known_intents_text()` iteriert über `REGISTRY` (sortiert nach
  Name) und hängt den Spezialfall `chat` (kein echter Command) an.
- Jeder Command kann optional ein `description`-Attribut setzen
  (jetzt ergänzt bei `OpenProgramCommand` und `ShutdownPcCommand`);
  fehlt es, wird nur der Name gezeigt - kein Zwang für neue Commands.
- `build_system_prompt()` wird bei jedem `get_plan()`-Aufruf neu
  gebaut statt einmalig als Modul-Konstante gecacht. Kosten sind
  vernachlässigbar (String-Join, kein API-Call), Vorteil: zur
  Laufzeit registrierte Commands sind sofort sichtbar.

## Alternativen erwogen
- **Statische Liste weiterpflegen, aber synchron halten:** verworfen -
  löst nicht das Grundproblem, dass README und Code auseinanderlaufen
  können; verlässt sich auf Disziplin statt auf Struktur.
- **Planner/AI-Interface entkoppeln (GPT-Vorschlag Punkt 4):**
  bewusst NICHT umgesetzt - aktuell nur ein AIEngine-Consumer
  (Planner), eine Abstraktionsschicht dafür wäre verfrühtes
  Overengineering (Regel 6, YAGNI). Wird erst relevant, wenn ein
  zweiter Aufrufer/Austausch der KI-Implementierung ansteht.

## Konsequenzen
- Neue Commands sind ab sofort ohne Änderung an `ai.py` in der Prompt-
  Liste sichtbar - README-Zusage stimmt jetzt tatsächlich.
- Phantom-Intents ohne Command sind aus dem Prompt verschwunden.
- Tests: `test_system_prompt_is_built_from_registry_not_hardcoded` und
  `test_system_prompt_includes_command_descriptions` in
  `tests/test_ai.py` prüfen genau das (inkl. Regression-Check, dass
  `search_google`/`weather` NICHT mehr im Prompt auftauchen).

## Falsifizierbarkeit
Gilt als richtig umgesetzt, wenn ein neuer Command mit `description`
automatisch im Prompt erscheint, ohne dass `core/ai.py` dafür
angefasst werden muss - Test dafür wäre ein neuer Dummy-Command, der
registriert und danach im Prompt gefunden wird (aktuell indirekt
abgedeckt, da `OpenProgramCommand`/`ShutdownPcCommand` schon über die
Registry statt hart codiert im Prompt erscheinen).
