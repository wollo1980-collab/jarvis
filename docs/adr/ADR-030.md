# ADR-030: Minimaler deterministischer Provider-Router in AIEngine (v0.8 Multi-KI, Phase 2)

## Kontext
v0.8 „Multi-KI" (Handbook Kap. 7/13) hat als End-Vision einen Orchestrator, der „je nach Aufgabe das beste Modell wählt" (Code → GPT, Review → Claude, lokal → Ollama). Phase 1 (ADR-029) hat dafür den ersten Baustein gelegt: die Provider-Abstraktion `LLMProvider` mit `OpenAIProvider` und `ClaudeProvider`, ausgewählt **explizit** über ein einziges Config-Feld `ai_provider` - für den gesamten Prozess genau ein Provider, kein Routing.

Phase 2 setzt den nächsten kleinsten Schritt darauf: Jarvis soll pro **Aufgabentyp** einen Provider verwenden können - deterministisch, ohne dass eine weitere KI die Wahl trifft. Product-Owner-Ziel dahinter: Jarvis soll möglichst bald die primäre KI-Oberfläche werden (ChatGPT-/Claude-Web seltener nötig), indem z. B. die freie Konversation auf das stärkere Modell gehen kann, während das günstige, JSON-stabile Intent-Parsing beim bisherigen Modell bleibt.

Ist-Zustand im Code (verifiziert):
- `AIEngine(config)` wird an drei Entry Points konstruiert (`main.py`, `jarvis_runtime.py`, `telegram_main.py`), immer nur mit `config`.
- `build_provider(config)` wird ausschließlich in `AIEngine.__init__` aufgerufen (`core/ai.py`).
- `get_plan()` hat genau einen Aufrufer: `core/planner.py` (Intent-Erkennung, braucht zuverlässiges JSON).
- `answer()` hat mehrere Aufrufer: `executor/executor.py` (freie Konversation) sowie `commands/monitor.py` und `commands/reports.py` (faktische Analyse). `answer()` ist damit **eine** Methode für mehrere Zwecke.

## Problemstellung
Wie kann Jarvis pro Aufgabentyp einen anderen Provider verwenden, **ohne**
- eine weitere KI/Heuristik-mit-LLM-Call für die Auswahl,
- eine Änderung der öffentlichen `AIEngine`-Schnittstelle oder ihrer vielen Aufrufer,
- die sicherheitskritische Logik (`confirmed`-Strip, JSON-Parsing, Fallbacks) zu duplizieren oder zu verschieben,
- die Lauffähigkeit von OpenAI-only-Setups (ohne installiertes `anthropic`) zu brechen?

Und wie bleibt das so klein, dass es ausschließlich den **Grundstein** legt, ohne einen intelligenten Orchestrator vorwegzunehmen?

## Entscheidung

### 1. Router als Backend-Weiche INNERHALB von AIEngine
`AIEngine` erhält einen kleinen, deterministischen `ProviderRouter` und eine Abbildung `dict[str, LLMProvider]` der **tatsächlich referenzierten** Provider (immer inklusive Standardprovider). `get_plan()`/`answer()` wählen intern anhand eines `TaskType` den zuständigen Provider aus. Der Router **konstruiert** nicht (das bleibt `build_provider`, ADR-029), sondern **wählt** pro Aufruf aus (Single Responsibility). Er ist die einzige neue, benannte Naht - genau dort kann eine spätere Phase feiner entscheiden.

`ProviderRouter` und `TaskType` werden in `core/providers.py` co-lokalisiert (kein neuer Modul-File für eine winzige Einheit; ein späteres Auslagern nach `core/routing.py` bleibt möglich, falls es wächst).

### 2. Öffentliche AIEngine-Schnittstelle bleibt unverändert
`get_plan(user_input, history)` und `answer(user_input, history, long_term_summary="")` behalten Signatur und Verhalten byte-identisch. Der `TaskType` wird **intern** gesetzt, nicht von außen übergeben. `AIEngine(config)` bleibt der einzige Konstruktor. Damit ändert sich nichts an `main.py`, `telegram_main.py`, `jarvis_runtime.py`, `core/planner.py`, `executor/*`, `commands/*` (monitor/reports) oder `reports/monitor.configure(ai)`.

### 3. Routing ausschließlich deterministisch, ohne zusätzlichen LLM-Aufruf
Die Providerwahl erfolgt über eine feste Nachschlagetabelle `TaskType → Provider-Name`. **Kein** Klassifikator, **kein** zusätzlicher Modell-Call, kein Zufall. Die Wahl darf **niemals** von Modell-Output beeinflusst werden (keine neue Trust-Boundary-Fläche); das einzige Auswahlsignal ist der intern gesetzte, vertrauenswürdige `TaskType`.

### 4. Genau zwei TaskTypes (Phase-2-Regeln)
```
get_plan()  -> TaskType.PLANNING
answer()    -> TaskType.GENERATION
```
Bewusst **nicht** geroutet wird nach **Intent** (der ist das *Ergebnis* von `get_plan()`, steht zum Aufrufzeitpunkt nicht zur Verfügung) und nicht nach **Sicherheitsstufe** (Eigenschaft des erst im Executor aufgelösten Commands, Kap. 10; Routing bleibt sicherheitsneutral). Dass `answer()` heute auch Analyse (monitor/reports) bedient, wird in Phase 2 bewusst **nicht** von der Konversation getrennt - beide teilen `TaskType.GENERATION`.

### 5. Config-Felder und Rückwärtskompatibilität
Zwei neue optionale Felder, gelesen wie die übrigen (Datei; Secrets weiterhin nur per Env, ADR-018):
```
planning_provider   # Provider für TaskType.PLANNING
answer_provider     # Provider für TaskType.GENERATION
```
`ai_provider` (ADR-029) bleibt der **Standardprovider** und damit Anker/Default/Fallback. Fehlt `planning_provider` oder `answer_provider`, verwendet der jeweilige TaskType automatisch `ai_provider`. Ist keines der beiden Felder gesetzt, verhält sich Jarvis **exakt wie heute** (alles auf `ai_provider`). Bestehende `config.json` funktionieren unverändert.

### 6. Fallback-Regeln
1. Der **Standardprovider (`ai_provider`) wird eager konstruiert** und muss erfolgreich initialisieren - andernfalls harter, früher Fehler (wie schon in Phase 1). Er ist der garantierte Anker.
2. **Nicht-Default-Provider bleiben lazy.** Ein Provider wird nur konstruiert, wenn eine Routing-Regel ihn referenziert. OpenAI-only-Setups (kein Routing auf `claude`) brauchen `anthropic` weiterhin nicht (Lazy-Import aus ADR-029 gilt fort).
3. Ist ein gerouteter Provider **nicht verfügbar** (Paket/Key fehlt) **oder wirft** sein `chat()`-Aufruf, erfolgt **ausschließlich für diesen einen Aufruf** ein Fallback auf den Standardprovider - mit WARNING-Log (kein stummer Wechsel).
4. Der Fallback greift **nur um den rohen `chat()`-Aufruf**. Danach laufen unverändert JSON-Parsing, der `confirmed`-Strip (Trust Boundary) und die bestehenden `get_plan`-/`answer`-Fallbacks (chat-Plan bzw. Fehlertext). Die **gesamte Sicherheits- und Formatlogik bleibt zentral in `AIEngine`**, providerunabhängig.

### 7. Logging (datensparsam)
Eine Zeile pro Aufruf, ohne Prompts, Antworten, API-Keys oder personenbezogene Inhalte:
- **INFO (Normalfall):** `TaskType`, gewählter Provider, Auswahlgrund (`regel` oder `default`).
- **WARNING (Fallback):** Providerfehler (Fehlerklasse, kein Inhalt) und Fallback auf den Standardprovider.
- Optional **DEBUG:** verwendetes Modell (für Kosten-/Latenz-Nachvollzug).

Die httpx/httpcore-Dämpfung auf WARNING (Token-Schutz) bleibt unberührt.

### 8. Ausdrücklich nur Grundstein, kein Orchestrator
Diese ADR hält ausdrücklich fest: Der Router ist eine **deterministische Weiche mit Nachschlagetabelle**, die lediglich den **Grundstein** für spätere, aufgabenabhängige Routing-Entscheidungen legt. Er ist **noch kein intelligenter Orchestrator**: er bewertet nichts, lernt nichts, ruft kein Modell zur Entscheidung, führt nichts parallel aus und trifft keine inhaltlichen Aufgaben-Klassifikationen. Die „beste Modell je Aufgabe"-Intelligenz der Handbook-Vision (Kap. 7) bleibt spätere Phase.

## Alternativen erwogen
- **Eigener Top-Level-Router über AIEngine (Aufrufer übergeben Provider/Router):** verworfen - zieht die Providerwahl aus `AIEngine` heraus, zwingt Aufrufer zur Änderung und verteilt die Sicherheits-Invariante. Widerspricht „Aufrufer unverändert".
- **Routing in `build_provider` (ProviderFactory):** verworfen - vermischt Konstruktion und Auswahl (SRP-Verletzung). Die Factory bleibt, der Router nutzt sie.
- **LLM-Klassifikator für die Providerwahl:** verworfen (Product-Owner) - zusätzlicher Call (Latenz/Kosten), neue Fehler-/Trust-Boundary-Fläche; nicht deterministisch/testbar.
- **Routing nach Intent oder Sicherheitsstufe in Phase 2:** verworfen - Intent ist Ergebnis von `get_plan()` (zum Call nicht verfügbar), Sicherheitsstufe erst im Executor bekannt; beides gehört frühestens in eine spätere Phase.
- **Optionaler `task`-Parameter an `answer()` zur Trennung Analyse/Konversation (`ANALYSIS`):** verschoben (Phase 3) - abwärtskompatibel möglich (Default-Wert), aber über den Phase-2-Bedarf hinaus (YAGNI).
- **Neues Modul `core/routing.py`:** verworfen für jetzt - zu wenig Code für einen eigenen File; Co-Location in `core/providers.py`.

## Konsequenzen
- `core/providers.py`: neuer `TaskType` (PLANNING, GENERATION) und ein kleiner, deterministischer `ProviderRouter` (Auswahl `TaskType → Provider-Name`, Fallback auf Standard).
- `core/ai.py`: `AIEngine` hält eine Provider-Map + den Router; `get_plan`/`answer` wählen intern per `TaskType` und wenden den Fallback um den `chat()`-Aufruf an. Prompt-Bau, JSON-Parsing, `confirmed`-Strip und die bestehenden Fallbacks bleiben unverändert.
- `core/config.py`: neue optionale Felder `planning_provider`, `answer_provider` (Default: leer → `ai_provider`); `config.example.json` dokumentiert sie.
- **Keine** Änderung an `main.py`, `telegram_main.py`, `jarvis_runtime.py`, `core/planner.py`, `executor/*`, `commands/*`, `memory/*`, `requirements.txt` (öffentliche Schnittstelle unverändert; keine neue Abhängigkeit).
- Bestehende Konfigurationen ohne die neuen Felder verhalten sich unverändert (reine Erweiterung).
- Umsetzung erst nach separater Product-Owner-Freigabe. Handbook-Übernahme (Provider-Router/Aufgaben-Routing) erst bei der nächsten Konsolidierung nach Abschluss der v0.8-Phasen.

## Risiken
1. **Kosten/Latenz:** `answer` auf ein stärkeres/teureres Modell (z. B. Claude Sonnet 5) erhöht Kosten/Latenz gegenüber `gpt-4o-mini` - bewusste Regelentscheidung, im Betrieb sichtbar per Logging.
2. **JSON-Zuverlässigkeit:** Routet man `PLANNING` auf Claude, gilt das Phase-1-Restrisiko (JSON nur per Prompt-Instruktion). Empfehlung/Default: `PLANNING` auf dem JSON-stabilen Provider belassen; das robuste Parsing/Fallback fängt Ausreißer weiterhin ab.
3. **Stiller Fallback maskiert Fehlkonfiguration:** Mitigation = deutliche WARNING-Logs bei jedem Fallback; kein stummer Wechsel.
4. **Persona-/Stilbruch innerhalb eines Gesprächs** (Plan von Provider A, Antwort von Provider B): geringfügige UX-Inkonsistenz, akzeptiert (nur `answer` ist nutzererlebt).
5. **Doppelte Provider-Konstruktion** nur bei tatsächlicher Referenzierung beider - dann werden beide SDKs/Keys benötigt; die Lazy-Regel verhindert Bruch bei Nicht-Nutzung.
6. **Config-Wildwuchs:** striktes Minimalschema (nur `planning_provider`/`answer_provider`), YAGNI - keine weiteren Routing-Achsen in Phase 2.

## Teststrategie
- **Router-Unit-Tests (deterministisch, kein Netz):** korrekte Auswahl je `TaskType`; `planning_provider`/`answer_provider` gesetzt vs. fehlend → Rückfall auf `ai_provider`; unbekannter Provider-Name → definierter Fehler/Fallback.
- **Fallback-Tests:** gerouteter Provider nicht verfügbar (Paket/Key fehlt) und gerouteter Provider wirft im `chat()` → beides fällt für den Aufruf auf den Standardprovider zurück; WARNING wird geloggt; Ergebnis kommt trotzdem zustande.
- **Rückwärtskompatibilität:** ohne neue Config-Felder identisches Verhalten wie heute (alles `ai_provider`); bestehende `test_ai.py`/`test_providers.py` bleiben grün.
- **Sicherheits-Regression (providerunabhängig):** der `confirmed`-Strip greift unabhängig vom gewählten Provider und auch nach einem Fallback; JSON-Parsing-Fallback unverändert.
- **Logging:** INFO enthält TaskType/Provider/Grund, WARNING den Fallback - Assertions prüfen, dass **keine** Prompt-/Antwort-/Key-Inhalte im Log stehen.
- SDK-Clients werden gemockt (wie in Phase 1); `anthropic` weiterhin über `sys.modules` simuliert (nicht installiert vorausgesetzt).

## Abgrenzung zu Phase 1 (ADR-029)
- Phase 1 lieferte **eine** Providerwahl für den gesamten Prozess (`ai_provider`), fest pro Lauf. Phase 2 erlaubt **pro Aufgabentyp** (`PLANNING`/`GENERATION`) einen Provider - weiterhin rein deterministisch und rein per Config.
- Phase 1 baute genau einen Provider (`self.provider`). Phase 2 baut eine kleine Provider-Map + Router; der Standardprovider bleibt der garantierte Anker/Fallback.
- Unverändert aus Phase 1: `LLMProvider`-Abstraktion, `build_provider` als Factory, Lazy/optionales `anthropic`, Secrets nur per Env, und - zentral - der `confirmed`-Strip sowie JSON-Parsing/Fallbacks in `AIEngine`.

## Bewusst NICHT Bestandteil von Phase 2
Intelligenter/lernender Orchestrator · LLM-basiertes Routing · Routing nach Intent · Routing nach Sicherheitsstufe · Trennung `ANALYSIS` vs. Konversation / optionaler `task`-Parameter an `answer()` · manuelles Laufzeit-Override („nutze Claude") · lokale Modelle/Ollama · MCP · RAG · Multi-Agent-System · paralleles/Fan-out-Ausführen mehrerer Modelle · Streaming · Kosten-/Latenz-Optimierungslogik · Änderung der öffentlichen `AIEngine`-Schnittstelle oder ihrer Aufrufer · Handbook-Übernahme (erst Konsolidierung nach v0.8).
