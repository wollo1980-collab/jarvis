# ADR-057: Anzeigename auf Zuruf ("nenn mich X")

## Status

**Accepted — PO-Freigabe am 2026-07-11.** 🟡 (neue Fähigkeit: Jarvis
schreibt erstmals zur Laufzeit an seiner eigenen `config.json`; Delegations-
Matrix CONTRIBUTING §3). Klein und in einem Stück umgesetzt.

## Problem/Kontext

Live-Befund 11.07.2026: Der PO hat Jarvis im Chat gesagt „nenn mich
Max". Der Chat übernahm es, aber die Dashboard-Begrüßung stand
weiter auf „PO". Ursache: Es gab **zwei Namensquellen**, die
auseinanderliefen.

- Dashboard-Begrüßung liest `config.owner_name` (`dashboard.py`).
- Chat-Prompt liest `config.owner_name` **und** die Fakten-Zusammenfassung
  des Langzeitgedächtnisses (`core/ai.py`).
- „nenn mich X" landete bisher nur als loser **Fakt** im Langzeitgedächtnis
  (`remember_fact`). Der Chat folgte zufällig dem Fakt, das Dashboard dem
  unveränderten Config-Wert → Drift.

Der PO-Wunsch: „Kann er das nicht automatisch mitändern, wenn ich merk dir
sage?" — der Name soll sich an EINER Stelle ändern und überall nachziehen,
ohne dass jemand von Hand nachträgt.

## Entscheidung

Ein **Namens-Wunsch aktualisiert `config.owner_name` direkt** — das eine
Feld, das Dashboard und Chat ohnehin beide lesen. Damit gibt es eine Quelle;
die Drift-Klasse verschwindet, nicht nur der Einzelfall.

1. **Erkennung (deterministisch, vor dem LLM).** `core/planner.py` erkennt
   ausdrückliche Selbst-Benennung per enger Regex-Liste („nenn mich X",
   „ich heiße X", „mein Name ist X", „sag X zu mir", „du darfst mich X
   nennen") und baut den Plan `set_owner_name`. Bewusst **eng**: Ein Fakt
   wie „Max ist mein Sohn" enthält kein Benennungs-Verb und fällt
   durch — er bleibt ein gewöhnlicher „merk dir"-Fakt. Eine Stoppwort-Liste
   fängt grammatische Treffer, die kein Name sind („nenn mich **nicht** so",
   „ich heiße dich **willkommen**"). Der LLM-Planner bleibt Fallback für
   ungewöhnliche Formulierungen (Command ist regulär registriert).

2. **Befehl `set_owner_name` (`commands/owner.py`).** Setzt `owner_name`
   im **live** geteilten Config-Objekt (der Chat folgt ab der nächsten
   Antwort, weil Runtime und AIEngine dasselbe Objekt halten) UND schreibt
   es via `persist_config_value` auf Platte.

3. **Persistenz (`core/config.py::persist_config_value`).** Aktualisiert
   genau einen Schlüssel in `config.json`, lässt alle anderen Werte und ihre
   Reihenfolge unberührt, atomar (tmp + `os.replace`). Bewusst schmal —
   aktuell nur für `owner_name`; kein Freibrief für beliebige Config-Mutation.

4. **Dashboard liest frisch (`dashboard.py::_live_owner_name`).** Das
   Dashboard läuft als **eigener Prozess** und bindet seine Config beim
   Start — ein In-Memory-Update im Runtime-Prozess erreicht es nicht. Es
   liest `owner_name` deshalb pro Status-Poll frisch aus `config.json`
   (Fallback auf die gebundene Config bei jedem Fehler, damit der Endpunkt
   nie bricht). Ergebnis: Die Begrüßung zieht binnen eines Poll-Zyklus
   (≤ 10 s) nach, **ohne Neustart**.

5. **Drift-Schutz.** Beim Setzen entwertet der Befehl etwaige alte
   Benennungs-**Fakten** im Langzeitgedächtnis (dieselben engen Muster),
   damit der Chat-Prompt nicht `owner_name` UND einen widersprechenden Fakt
   sieht. Neue Namenswünsche erzeugen keinen Fakt mehr (sie gehen an
   `set_owner_name`, nicht an `remember_fact`).

## Sicherheitsmodell / Governance

Das Neue ist **Config-Schreiben zur Laufzeit** — deshalb diese ADR (🟡).
Die Schwelle ist einmalig: Ab jetzt ist eine Namensänderung reiner Betrieb
(„nenn mich X" → fertig), keine weitere ADR/Freigabe je Nutzung. Die
Fähigkeit ist eng: Sicherheitsstufe 1 (lokal, umkehrbar — einfach neu
benennen), schreibt ausschließlich `owner_name`, kein Secret, keine
Systemaktion. `persist_config_value` ist absichtlich single-key und wird
nur mit `"owner_name"` aufgerufen; eine Ausweitung auf andere Felder wäre
eine neue Entscheidung.

## Alternativen

- **Nur `config.json` schreiben + Neustart:** verworfen — Neustart-Flackern
  für eine Namensänderung ist unverhältnismäßig; Frisch-Lesen ist billiger.
- **Namen als eigene „Preferences"-Datei (neue Quelle der Wahrheit):**
  verworfen — `owner_name` ist bereits die gemeinsame Quelle; eine zweite
  Quelle hätte die Drift nur verlagert.
- **Weiter nur als Fakt speichern, Dashboard den Fakt parsen lassen:**
  verworfen — dann parst das Dashboard Freitext und die zwei Quellen bleiben.

## Falsifizierbarkeit

Gilt als falsch geschnitten, wenn die enge Erkennung in der Praxis echte
Namenswünsche regelmäßig verpasst (dann Muster erweitern) ODER umgekehrt
Fakten fälschlich zur Anrede macht (dann Stoppwort-/Verb-Bindung
verschärfen). Beides ist über Tests und den Live-Gebrauch beobachtbar.
