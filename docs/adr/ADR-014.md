# ADR-014: Excel-Lesen (v0.5 Phase 1) - openpyxl, flache Struktur, kein ai.py-Sonderfall

## Problem
ADR-013 (Handbook v3.3) legt Scope und Sicherheitsstufen für den Excel-Baustein fest, verlangt aber vor jeglichem Code einen technischen Vorschlag (Bibliothek, Commands, Registry-Integration). Dieser wurde erstellt und von Wolfgang mit einer Präzisierung freigegeben: **kein neuer command-spezifischer Sonderfall in `core/ai.py`** - anders als bei `remember_fact`/`forget_fact` (ADR-009), die dort einen eigenen Prompt-Block bekommen haben.

## Entscheidung
Neuer Command `commands/excel.py::ReadExcelCommand` (Intent `read_excel`, Sicherheitsstufe 0 laut Handbook Kap. 10 v3.3 - `requires_confirmation = False`):

- **Bibliothek: `openpyxl`**, `load_workbook(path, read_only=True, data_only=True)`. Kein `pandas` (zu schwere Abhängigkeit für reines Lesen, YAGNI), kein `xlrd` (unterstützt seit v2.0 kein `.xlsx` mehr), kein `win32com`/echtes Excel per COM (bräuchte eine installierte/laufende Excel-Instanz für eine Sicherheitsstufe-0-Leseaktion - unverhältnismäßig).
- **Format-Scope:** nur `.xlsx`/`.xlsm`. Kein `.xls` (Legacy, von `openpyxl` nicht unterstützt), kein CSV.
- **Kein KI-Sonderfall in `core/ai.py`:** Anders als bei den Memory-Commands beschreibt `ReadExcelCommand.description` selbst ausführlich, wie `target` (vollständiger Dateipfad) und optional `parameters.sheet` zu befüllen sind. Der bestehende Registry-Mechanismus (`_known_intents_text()`, ADR-007) reicht dafür aus - verifiziert durch direkten Aufruf von `build_system_prompt()`, der `read_excel` inkl. der vollen Beschreibung enthält, ohne dass `core/ai.py` geändert wurde.
- **Architektur bleibt flach** (bestätigt ADR-013/Kap. 30 v3.3): `commands/excel.py` nach demselben Rezept wie `monitor.py`/`installer.py`, kein `configure()`-Mechanismus nötig (zustandslos), keine Migration auf `tools/office/...`.
- **Kein Datei-Katalog** wie `KNOWN_PROGRAMS`/`KNOWN_PACKAGES`: Dateipfade sind frei/beliebig (Reports wechseln), kein fester kleiner Katalog wie bei Programmen/Paketen sinnvoll. Direkt in der Spracheingabe genannter Pfad reicht für Phase 1 (keine bekannten Report-Pfade, kein Memory-Automatismus - explizite Entscheidung von Wolfgang).
- **Keine KI-Zusammenfassung im Command:** `execute()` liefert einen deterministischen Text (Blattnamen + Dimensionen) plus die gelesenen Rohdaten in `Result.data["sheets"]`. Inhaltliche Zusammenfassung bleibt einem späteren Baustein (Tabellen-Auswertung) überlassen, der auf diesen Rohdaten aufbaut.
- **Zwei benannte Obergrenzen** (keine Magic Values, Kap. 5): `_MAX_ROWS_PER_SHEET = 500` begrenzt sowohl die tatsächlich eingelesenen als auch zurückgegebenen Daten pro Blatt (Schutz vor unbegrenztem Speicherverbrauch bei sehr großen Dateien), `_PREVIEW_ROWS_IN_MESSAGE` ist als Konstante vorgesehen, wird in der aktuellen Nachricht (nur Blattnamen + Dimensionen) noch nicht gebraucht, aber für eine spätere feinere Textvorschau bereitgehalten.
- **`workbook.close()` in `finally`:** read-only Workbooks halten sonst einen offenen Dateihandle - unter Windows relevant, falls die Datei danach in Excel geöffnet werden soll.

## Begründung
- Sicherheitsstufe 0 (reines Lesen) passt exakt zur Handbook-Vorgabe (Kap. 10 v3.3).
- Ausführliche `description` statt `ai.py`-Sonderfall hält den Registry-Mechanismus (ADR-007) konsistent: neue Commands sollen möglichst ohne Anfassen von `ai.py` auskommen. Bei `remember_fact`/`forget_fact` war ein Sonderfall nötig, weil dort eine Kategorisierung (`parameters.category`) mit einer festen Werteliste erklärt werden musste, die sich nicht aus einer normalen Beschreibung ergibt - bei `read_excel` reicht eine gute Beschreibung, da Dateipfad und optionales Sheet selbsterklärend sind.
- Rohdaten statt KI-Zusammenfassung im Command hält den Scope eng (Phase 1 = Lesen) und vermeidet, dass ein zukünftiger Tabellen-Auswertung-Baustein an eine bereits von der KI "vorverdaute" Zusammenfassung gebunden ist statt an die eigentlichen Zelldaten.
- Zeilen-Obergrenze verhindert, dass eine sehr große Tabelle unbegrenzt Speicher verbraucht oder den Chat flutet.

## Alternativen erwogen
- **`pandas` als Lese-Engine:** verworfen für Phase 1 - deutlich schwerere Abhängigkeit ohne aktuellen Bedarf (keine Datenanalyse in dieser Phase), YAGNI.
- **KI-Zusammenfassung direkt im Command (zweiter `AIEngine`-Aufruf wie bei `answer()`):** verworfen - würde den Scope von "reines Lesen" aufweichen und ist explizit dem späteren Tabellen-Auswertung-Baustein vorbehalten (Reihenfolge laut Wolfgang: Excel → Tabellen-Auswertung → KPI → Power BI).
- **Bekannte/gemerkte Report-Pfade (Memory-Integration):** verworfen für Phase 1 - explizite Entscheidung von Wolfgang, Dateipfad direkt in der Spracheingabe reicht.

## Konsequenzen
- Neuer Intent `read_excel`, automatisch im KI-Prompt sichtbar über die Registry (ADR-007-Mechanismus), keine Änderung an `core/ai.py`, `planner.py`, `tool_manager.py` oder `executor.py` nötig.
- `openpyxl` ist ab sofort eine feste Abhängigkeit (`requirements.txt`).
- Schreiben/Formatieren/Power Query/Makros sowie `.xls`-Support bleiben offen für eine spätere, eigene Priorisierung.
- Tabellen-Auswertung (nächster Baustein laut Wolfgangs Reihenfolge) kann auf `Result.data["sheets"]` aufbauen.
