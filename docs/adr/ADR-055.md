# ADR-055: Verifikations-Harnisch - Jarvis prueft die Arbeit selbst

## Problem/Kontext

PO-Ziel (11.07.2026, [Angestellten-Vision], Kampagne B): Jarvis soll so
gut werden wie Claude jetzt - und ein zentraler Baustein davon ist die
**Selbstkontrolle**. Heute baut ein Agent im Kaefig (ADR-050), kann aber
aus Sicherheitsgruenden nichts ausfuehren - er dokumentiert eine ehrliche
"Verifikationsluecke". Diese Luecke fuellt bislang ein Mensch (der Lead
Engineer fuehrt pytest + Gate aus). Stufe 3 heisst: Jarvis fuellt sie
selbst und legt einen Pruefbericht vor.

## Entscheidung

Ein **Verifikations-Harnisch** (`core/verify.py` + `commands/verify.py`,
Intent `verify_repo`), der in EINEM freigegebenen Repo ein HART
VERDRAHTETES Befehls-Whitelist laufen laesst und einen strukturierten
Bericht liefert:

1. **Konsistenz-Gate:** `<python> scripts/check_consistency.py` (falls im
   Repo vorhanden, sonst uebersprungen).
2. **Testsuite:** `<python> -m pytest -q --basetemp <frisch>`.

## Sicherheitsmodell (der springende Punkt)

Der Harnisch gibt Jarvis KEINE freie Kommando-Ausfuehrung:

- **Feste argv-Listen:** Jeder Befehl wird im Code als Liste gebaut, nie
  aus Nutzer- oder Modell-Eingabe. Es gibt kein "fuehre X aus".
- **Kein Shell:** `subprocess.run(..., shell=False)` - keine Shell-
  Interpolation, keine Verkettung.
- **Repo-Allowlist (fail-closed):** Die einzige Variable ist der Repo-Pfad;
  er kommt aus einer Allowlist (`agent_repos` + `agent_write_repos`,
  validierte Pfade). Ein nicht gelistetes Repo ist nicht pruefbar. Der
  Alias aus der Eingabe waehlt nur AUS der Allowlist, er bildet nie einen
  Pfad.
- **Nur eigene Repos:** Die Allowlist enthaelt ausschliesslich Repos, die
  wir kontrollieren (jarvis, jkc). Deren pytest ist unser eigener Code -
  keine Ausfuehrung fremden Codes.
- **Kein Schreiben am Quellcode, kein Commit:** Der Harnisch liest, laesst
  laufen, berichtet. `requires_confirmation=False` ist damit gerechtfertigt
  (Stufe 0/1). Das Committen bleibt beim PO (unveraendert).
- **Wall-Clock-Timeout + gekappte Ausgabe:** kein Haenger, keine Flut.
- **Fail-safe:** wirft nie; ein fehlender/fehlgeschlagener Befehl wird
  berichtet, nicht als Absturz.

## Begruendung

Die riskante Faehigkeit waere "Jarvis fuehrt beliebige Befehle aus". Genau
die entsteht hier NICHT: die Pruefung bekommt zwei feste Werkzeuge, mehr
nicht. Der Agent im Kaefig behaelt seine Read-only-/Write-Cage-Grenzen -
der Harnisch ist ein getrennter, ebenso enger Schritt DANACH. Das Ergebnis
ist rein informativ (Bericht), loest nie selbst eine Aktion aus (Trust
Boundary wie delegate_analysis).

## Alternativen

- **Freies Shell fuer die Pruefung:** verworfen - das ist die eine Sache,
  die wir nicht wollen (beliebige Ausfuehrung).
- **Verifikation weiter manuell (Mensch):** verworfen - genau die Luecke,
  die Stufe 3 schliessen soll.
- **Gate/Tests im selben Prozess importieren statt Subprozess:** verworfen -
  ein frischer Subprozess isoliert sauber (eigener Import-Zustand, eigenes
  basetemp) und kann die live laufende Runtime nicht stoeren.

## Konsequenzen

- Neuer Intent `verify_repo` (long_running -> Runtime fuehrt async aus, die
  Suite dauert). Vorerst NICHT ueber Telegram exponiert (bewusst
  konservativ; kann spaeter folgen).
- Fundament fuer die naechsten B-Scheiben: den Harnisch nach einer
  Kaefig-Schreib-Delegation automatisch anstossen und den Bericht in die
  Delegations-Antwort aufnehmen (dann ist die Kette Bauen->Pruefen->
  Vorlegen geschlossen, Commit bleibt PO).

## Falsifizierbarkeit

Gilt als falsch dimensioniert, wenn sich zeigt, dass Gate + pytest nicht
ausreichen, um die Verifikationsluecke ehrlich zu schliessen (z. B. Repos
mit anderem Testrunner) - dann Review dieses ADRs vor Erweiterung des
Whitelists; niemals stillschweigend freies Shell zulassen.

## Bewusst NICHT

- Kein freies/aus Eingabe gebildetes Kommando.
- Kein Schreiben am Quellcode, kein Commit, kein git.
- Keine Repos ausserhalb der validierten Allowlist.
