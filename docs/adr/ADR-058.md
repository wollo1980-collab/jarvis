# ADR-058: Spotify-Connector (Sprachsteuerung der eigenen Wiedergabe)

## Status

**Accepted — PO-Freigabe am 2026-07-11** (🟡, neuer Connector/Integration +
externe Auth mit Secrets, CONTRIBUTING §3). PO-Wunsch „Spotify wäre nice" +
Bestätigung Spotify Premium vorhanden.

## Problem/Kontext

Jarvis soll die Musikwiedergabe des Nutzers per Sprache/Chat steuern können
(„spiel die Playlist Fokus", „pause", „nächster Song", „was läuft gerade?").
Spotify bietet dafür die Web API — aber sie erfordert OAuth (kein reiner
API-Key) und, für **Wiedergabe-Steuerung**, ein **Premium**-Konto sowie ein
**aktives Abspielgerät** (Spotify Connect). Jarvis ist selbst kein
Abspielgerät; er dirigiert ein laufendes Spotify.

## Entscheidung

Ein schlanker Connector `core/spotify.py` + Sprachbefehle `commands/spotify.py`
nach dem etablierten Muster (keyless-Wetter/News: Client in `core/`, Command in
`commands/` mit `configure()`, HTTP-Schicht injizierbar für Tests).

- **Auth = Authorization-Code-Flow, EINMALIG.** `scripts/spotify_auth.py` holt
  per einmaligem Browser-Login einen `refresh_token` (lokaler Loopback-Callback
  auf `127.0.0.1`, CSRF-`state`-Prüfung). Zur Laufzeit tauscht der Client den
  `refresh_token` gegen kurzlebige `access_token` — **kein weiterer Login**.
- **Secrets in der Config** (`spotify_client_id`, `spotify_client_secret`,
  `spotify_refresh_token`), env-überschreibbar. `config.json` ist gitignoriert +
  vom `release_scan`-Türsteher abgesichert — nichts davon ins Repo.
- **Fähigkeiten:** abspielen/fortsetzen (mit Suche nach Playlist/Song), pausieren,
  nächster/vorheriger Titel, Lautstärke (absolut + relativ), „was läuft gerade?".
- **Fail-closed:** fehlen Credentials, ist die Fähigkeit aus; die Befehle
  antworten ehrlich „nicht eingerichtet". Kein Absturz.

## Sicherheitsmodell

Sicherheitsstufe 0 (wie Wetter/News), bewusst **ohne** Bestätigungs-Gate:
Wiedergabe-Steuerung betrifft ausschließlich die **eigene** Musik des Nutzers,
ist **reversibel** (pause = undo play) und löst keine nach-außen-Wirkung aus
(kein Senden, kein Löschen, kein Geld). Das ist der Grund, warum es — anders als
Mail-Senden — kein 🔴/Gate braucht. Die OAuth-Scopes sind auf genau das
Nötige begrenzt (`user-read-playback-state`, `user-modify-playback-state`,
`user-read-currently-playing`, `playlist-read-private`).

## Alternativen

- **Kein OAuth, nur API-Key:** nicht möglich — Spotify verlangt OAuth.
- **PKCE statt Client-Secret:** ginge, aber der lokale Helfer ist ein
  vertraulicher Client; Client-Secret + Refresh-Token ist der einfachere,
  gut dokumentierte Weg. PKCE bleibt als spätere Verfeinerung offen.
- **Lokale Steuerung (Desktop-App-Fernsteuerung) statt Web API:** brüchig,
  plattformabhängig; die Web API ist der stabile, dokumentierte Weg.

## Konsequenzen

- Neue optionale Abhängigkeit: **keine** (stdlib-only, `urllib`).
- Braucht Premium + aktives Gerät — ehrlich in README/Fehlermeldungen benannt
  (`NoActiveDeviceError` → „starten Sie Spotify irgendwo").
- Der `refresh_token` ist langlebig; wird er ungültig (Passwortwechsel,
  Rechte-Entzug), meldet der Client `SpotifyAuthError` → Neu-Einrichtung.

## Bewusst NICHT (in dieser ADR)

- Keine Bibliotheks-/Playlist-Verwaltung (anlegen/löschen/Songs hinzufügen).
- Keine geräteübergreifende Auswahl/Transfer (immer das aktive Gerät).
- Keine Telegram-Fernsteuerung (kann später ergänzt werden, eigener Schritt).

## Falsifizierbarkeit

Gilt als falsch geschnitten, wenn sich zeigt, dass der Refresh-Token-Weg im
unbeaufsichtigten Betrieb regelmäßig bricht (dann PKCE/erneuter Flow nötig),
oder wenn „aktives Gerät" in der Praxis so oft fehlt, dass die Fähigkeit nervt
statt hilft — dann Review vor Ausbau.
