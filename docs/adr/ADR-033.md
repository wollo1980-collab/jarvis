# ADR-033: Modellunabhängiger Delegationsprozess (Agenten-Orchestrierung)

## Problem/Kontext

Die Produktvision (PO, 2026-07-06) präzisiert Jarvis als **Vermittlungsschicht**
über drei Ebenen — Informationen (Outlook/Teams/Kalender/Browser/OneDrive/
GitHub), **Agenten** (Claude/Codex/GPT/künftige) und Geräte (PC/NAS/Smartphone/
Smart Home). Der als Erstes gewählte Ausbaupfad ist der **Agenten-Arm**:
Jarvis soll Arbeit an spezialisierte Agenten delegieren (z. B. „während ich
unterwegs bin: Repo X analysieren, einen Fix vorbereiten, für mein Review
bereitlegen").

Bevor ein konkreter Agent gewählt oder eine Fähigkeit gebaut wird, braucht es
einen **modellunabhängigen Orchestrierungsprozess**, auf den alle Agenten
aufsetzen. Sonst wird Delegation pro Agent neu erfunden, Agenten werden zum
Lock-in statt austauschbaren Backends (verletzt die Modellneutralitäts-
Invariante), und die Sicherheit bliebe uneinheitlich. Delegation an Agenten ist
zugleich die **höchste Vertrauens-/Sicherheitsfläche** des Systems
(unbeaufsichtigte Ausführung, ferngesteuert) und muss deshalb in das bestehende
Sicherheits- und Governance-Modell eingebettet sein, nicht daneben stehen.

## Entscheidung

Jede Delegation durchläuft einen festen, modellunabhängigen Lebenszyklus:

1. **Erfassen** — Wunsch in natürlicher Sprache, kanalunabhängig.
2. **Verstehen & Einstufen** — strukturierter Auftrag (Ziel · Ziel-Ressource ·
   erwartetes Ergebnis) + Sicherheitsstufe (lesen/schreiben/ausführen) +
   Zulässigkeit (Ressourcen-Allowlist, erlaubter Remote-Scope). Unklar oder
   unzulässig → fail-closed, Rückfrage statt Raten.
3. **Priorisieren & Schneiden** — große oder unklare Wünsche in kleine,
   reviewbare Arbeitspakete zerlegen; nie als Großauftrag delegieren. Ist der
   Wunsch unklar, wird er zuerst mit dem Menschen geschärft. Jedes Teilpaket ist
   einzeln einstufbar, delegierbar und reviewbar.
4. **Zuordnen (modellneutral)** — benötigte *Fähigkeit* bestimmen, passendes
   Agenten-Backend wählen. Der Prozess kennt Fähigkeiten, keine Modellnamen.
5. **Eingrenzen & Freigeben** — Arbeitsrahmen *vor* der Ausführung setzen
   (Branch/Worktree, read-only vs. write, Zeit-/Kosten-/Token-Limit, klare
   Tabus: nie `main`, nie Push, nichts Irreversibles); Freigabe-Gate je Stufe.
   Trust Boundary: der Mensch autorisiert das Handeln, nie der Agenten-Output
   sich selbst.
6. **Ausführen & Beobachten** — Agent im Sandbox-Rahmen starten, jede Aktion
   protokollieren, Limits hart durchsetzen (Timeout, Kostendeckel, Kill-Switch).
7. **Verifizieren & Bewerten** — Ergebnis objektiv prüfen wo möglich
   (Tests/Gate laufen lassen) statt dem Agenten-Wort zu vertrauen; Status
   ✓ / ✗ / ? vergeben.
8. **Zurückgeben & Übergeben** — als Arbeitspaket verpacken (Zusammenfassung ·
   Artefakte · Status · nächste Schritte); nichts Irreversibles ohne
   menschliches Review. Freigabe/Commit/Merge bleibt beim Menschen (zeitversetzt
   bei Rückkehr).
9. **Protokollieren & Abschließen** — vollständige Nachvollziehbarkeit (Auftrag ·
   Einstufung · Agent · Aktionen · Kosten · Ergebnis). Geschlossen erst, wenn das
   Ergebnis angenommen oder verworfen wurde.

**Durchlaufende Invarianten:** Modellneutralität (Auswahl über Fähigkeit, nicht
Modell) · Sicherheitsstufen gelten unverändert · Trust Boundary · der Mensch
bleibt souverän (Ergebnis ist *vorbereitet*, nie selbsttätig übernommen) ·
Nachvollziehbarkeit/Observability · fail-closed.

**Einordnung:** Der Prozess ist die bestehende Pipeline
(`Eingabe → Planner → Tool Manager → Executor (✓/✗/?)`), verallgemeinert — ein
**Agent ist eine neue Tool-Klasse**, kein Parallelsystem. Evolution statt
Revolution.

## Begründung

- **Modellneutralität (Invariante):** Agenten bleiben austauschbare Backends;
  Delegation wird nicht pro Agent neu gebaut.
- **Sicherheit an der richtigen Stelle:** Die höchste Vertrauensfläche des
  Systems erbt das bewährte Sicherheitsmodell (Stufen, Trust Boundary, fail-
  closed) statt eines Sonderwegs.
- **Kleine, reviewbare Pakete** sind das Kernprinzip des Projekts — jetzt auf
  Agentenarbeit angewandt; sie ermöglichen erst das zeitversetzte Review.
- **Mensch bleibt souverän:** Ergebnisse werden vorbereitet, nie eigenmächtig
  übernommen.

## Alternativen erwogen

- **Pro Agent eigene Integration ohne gemeinsamen Prozess:** verworfen —
  Lock-in, Duplizierung, uneinheitliche Sicherheit.
- **Großauftrag direkt an einen Agenten:** verworfen — unreviewbar, unsicher,
  Agenten laufen bei vagen Großaufträgen aus dem Ruder.
- **Sofort einen Agenten wählen und bauen:** verworfen — Prozess vor Umsetzung
  (ausdrücklicher PO-Wunsch); der Prozess muss modellunabhängig stehen, bevor
  ein Backend gewählt wird.

## Konsequenzen

- Dieser ADR legt **nur den Prozess** fest — **keinen** konkreten Agenten,
  **keine** erste Fähigkeit, **keinen** Code. Beides folgt in einem Aufsatz-ADR
  (erste Scheibe: read-only Analyse), inklusive des dann zu definierenden
  **Agenten-Backend-Kontrakts** (Protokoll analog `LLMProvider`).
- Der Prozess ist Kandidat für eine **Architektur-Invariante im Handbook**
  (🔴, später), sobald er an einem realen Baustein bewährt ist.

## Falsifizierbarkeit

Zu revidieren, wenn sich der Prozess bei der ersten echten Delegation als
unpraktikabel oder unvollständig zeigt (ein Schritt fehlt, behindert oder wird
in der Praxis übersprungen), oder wenn die Modellunabhängigkeit nicht hält
(ein Backend erzwingt Prozessänderungen).

## Bewusst NICHT in diesem Schritt

Agentenwahl (Claude/Codex/GPT) · erste Fähigkeit/Scheibe · Agenten-Backend-
Protokoll als Code · Multi-Agenten-Orchestrierung · Geräte-Ebene (NAS/Smart
Home) · Handbook-Invariante · jegliche Implementierung.
