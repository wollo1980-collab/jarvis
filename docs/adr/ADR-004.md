# ADR-004: Planner / Tool Manager / Executor als eigene Schicht

## Problem
v0.2 hatte nur main.py -> ai.get_plan() -> commands.dispatch(). Das
deckte genau einen Intent pro Nutzereingabe ab, hatte keinen Ort für
eine Bestätigung vor kritischen Aktionen (Trockenlauf-Prinzip, Kap. 10)
und keine einheitliche ✓/✗/?-Ergebnismeldung (Kap. 5).

## Entscheidung
Drei neue, kleine Bausteine statt eines großen Rewrites:

- `core/planner.py::Planner` - zerlegt die Nutzereingabe naiv an
  "und" / "und dann" / "danach" / ";" in 1..n Teilsätze und holt für
  jeden Teilsatz einen Plan von der (unveränderten) `AIEngine`.
- `core/tool_manager.py::ToolManager` - schlägt den passenden Command
  in der bestehenden `commands.REGISTRY` nach. Reine Lookup-Funktion,
  keine Ausführung.
- `executor/executor.py::Executor` - führt die Schritte der Reihe
  nach aus, holt bei `requires_confirmation`-Commands eine Bestätigung
  ein, generiert bei chat-Intents über `AIEngine.answer()` eine echte
  Antwort, und bricht bei Fehlern/offenen Rückfragen ab statt einfach
  weiterzumachen.

`commands.dispatch()` bleibt unverändert bestehen (Executor nutzt es
weiterhin für den chat-Fallback und für unbekannte Intents) - kein
Bruch an bestehendem Code, kein Rewrite von main.py notwendig.

## Begründung
- Separation of Concerns (Kap. 4/31): "welches Tool passt" (Tool
  Manager) ist eine andere Frage als "das Tool ausführen" (Executor).
- Regel 6 (keine Architecture Astronautics): die Eingabe-Splitting-
  Heuristik ist bewusst naiv (Regex statt eigenem KI-Aufruf für Multi-
  Step-Zerlegung) - deckt den heutigen Bedarf (2-3 Aktionen pro Satz).
- Testbarkeit (Kap. 4): Planner, Tool Manager und Executor sind je
  einzeln mit Mocks testbar, ohne main.py oder einen echten API-Key
  zu benötigen (siehe tests/test_planner.py, test_executor.py).

## Konsequenzen
- Zwei API-Calls pro chat-Turn (Intent-Erkennung + Antwortgenerierung)
  statt einem - höhere Kosten/Latenz. Siehe Falsifizierbarkeit unten.
- Der Split-Regex trennt keine echten Multi-Step-Pläne mit
  Abhängigkeiten zwischen Schritten (z. B. "wenn X erfolgreich war,
  dann Y") - das bleibt ein Later-Feature (Kap. 27).

## Falsifizierbarkeit
Gilt als unzureichend, wenn:
- Nutzer regelmäßig zusammengesetzte Sätze verwenden, die der
  Split-Regex falsch trennt (Review in v0.4).
- Die Latenz/Kosten durch den zweiten API-Call (get_plan + answer)
  spürbar stören (dann: ein kombinierter Aufruf in v0.4 prüfen).
