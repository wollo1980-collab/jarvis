# ADR-013: Handbook v3.3 - Excel-Baustein (v0.5) Scope, Sicherheitsstufen, Governance

## Problem
Eine Handbook-Pruefung (v3.2) vor Beginn von `v0.5 "Arbeitsmodule"` deckte mehrere Luecken auf:

1. Kap. 13 (Roadmap) nennt fuer `v0.5` nur pauschal "Tabellen-Auswertung, KPI, erweiterte Auswertungen, Excel" - ohne Phasenabgrenzung. Unklar war, ob "Excel" Lesen UND Schreiben (inkl. Power Query, Makros, Formatierung) meint.
2. Kap. 27 fasst "Excel- und Outlook-Integration" als einen Punkt zusammen, obwohl es zwei unabhaengige, getrennt zu priorisierende Bausteine sind.
3. Kap. 10 (Sicherheitsstufen) enthielt keine expliziten Eintraege fuer Datei-Lese-/Schreib-/Loeschzugriffe (z. B. Excel), nur allgemeine Beispiele.
4. Kap. 12 zeigt `tools/office/excel.py`/`outlook.py` als "Zielstruktur ab v0.5+" - unklar, ob das bedeutet, dass v0.5 automatisch dorthin umstrukturiert werden muss, obwohl Kap. 30 nur "Architektur erst bei v0.5+ pruefen" sagt.
5. Kap. 19 "Logbook" hatte in v3.2 keinen inhaltlichen Text, obwohl in v0.4 bereits zwei neue Governance-Dokumente (`docs/AI_START.md`, `docs/PROJECT_STATE.md`, siehe ADR-010) eingefuehrt wurden, die im Handbook nicht erwaehnt waren.
6. Kap. 28 (Definition of Done) hatte keine v0.4- oder v0.5-spezifischen Kriterien, nur allgemeine plus v0.2-/v0.3-spezifische.

PO (Product Owner) hat dazu eine Handbook-Pruefung angestossen (inkl. Rueckfrage an einen externen Mentor/"GPT") und vor jeglicher Excel-Implementierung folgende Entscheidungen getroffen.

## Entscheidung
1. **Excel-Scope v0.5 = Phase 1, nur Lesen.** Datei oeffnen, Arbeitsblaetter erkennen, Tabellen/Zellen lesen, Inhalte zusammenfassen. Ausdruecklich NICHT Teil von Phase 1: Schreiben, Formatieren, Power Query, Makros, Formel-Bearbeitung.
2. **Architektur bleibt flach.** Keine Migration auf die Kap.-30-Zielstruktur (`tools/office/...`) fuer v0.5. Ein neuer Excel-Command wird wie gewohnt unter `commands/` ergaenzt (bestehendes Rezept, siehe README "Neuen Command hinzufuegen"). Umstellung auf `tools/office/...` erst, wenn mehrere Office-Module tatsaechlich existieren (Regel 6, YAGNI) - nicht automatisch mit v0.5.
3. **Sicherheitsstufen fuer Dateizugriffe ergaenzt** (Kap. 10): Excel lesen = Stufe 0 (keine Bestaetigung), Excel schreiben = Stufe 2 (Bestaetigung erforderlich), Datei loeschen = Stufe 3 (mehrfache Bestaetigung, exakte Phrase wie bei `shutdown_pc`).
4. **Technischer Vorschlag vor Code Pflicht.** Vor jeglicher Excel-Implementierung muss ein kurzer technischer Vorschlag (Bibliothekswahl, betroffene Commands, Registry-Integration) erstellt und vom Product Owner ausdruecklich freigegeben werden. Es wird noch KEIN Code geschrieben.
5. **Outlook ist nicht Teil von v0.5.** Excel und Outlook werden getrennt priorisiert (Kap. 27 entsprechend praezisiert).
6. **Handbook v3.2 -> v3.3.** Da das Handbook laut eigener Konvention (Kap. 2, neu ergaenzt) nur zwischen zwei Jarvis-Versionen geaendert werden darf, werden alle obigen Praezisierungen jetzt (vor Beginn von v0.5, nach Abschluss von v0.4) als Handbook v3.3 nachgezogen - nicht waehrend der v0.5-Implementierung.
7. **Governance-Dokumente offiziell im Handbook verankert** (Kap. 19): `docs/AI_START.md`, `docs/PROJECT_STATE.md` und das ADR-System (bereits seit ADR-010 gelebt) werden jetzt auch im Handbook selbst erwaehnt, statt nur in README/ADR-010.
8. **Definition of Done um v0.4- und v0.5-spezifische Kriterien ergaenzt** (Kap. 28), analog zu den bereits vorhandenen v0.2-/v0.3-spezifischen Abschnitten.

Alle Aenderungen sind in `docs/handbook/JARVIS_MASTER_HANDBOOK_v3_3.docx` umgesetzt. `v3_2.docx` bleibt unveraendert als historisches Dokument erhalten (Grundlage fuer v0.4).

## Begruendung
- Ohne Phasenabgrenzung waere "Excel-Integration" ein Einladung zu Scope Creep (Schreiben/Power Query/Makros sind erheblich riskanter und aufwendiger als Lesen) - Stufe-Trennung folgt dem in ADR-006 gelernten Prinzip, Sicherheitsstufen nicht zu vermischen.
- Flache Architektur fuer ein einzelnes Modul vermeidet Architecture Astronautics (Regel 6) - eine Umstrukturierung lohnt sich erst bei mehreren Office-Modulen.
- Ein Freigabe-Schritt vor Code passt zum bestehenden Trockenlauf-Prinzip (Kap. 10) und zur Produkt-Owner-Regel ("KI darf Umsetzung vorschlagen, aber keine Roadmap aendern").
- Handbook-Aenderungen nur zwischen Versionen (statt mitten in v0.5) haelt eindeutig nachvollziehbar, auf welchem Handbook-Stand welcher Code basiert - direkte Konsequenz aus der in dieser Pruefung entdeckten Unklarheit.

## Alternativen erwogen
- **Excel-Lesen-und-Schreiben in einem Rutsch fuer v0.5:** verworfen. Deutlich hoeheres Risiko (Datenverlust bei Schreibzugriff), keine Notwendigkeit fuer die von PO genannten Anwendungsfaelle (Tabellen-Auswertung/KPI lesen reicht fuer den Start).
- **Sofortige Umstellung auf `tools/office/...`:** verworfen. Kap. 30 verlangt nur eine Pruefung "ab v0.5+", nicht eine sofortige Umsetzung - mit nur einem Office-Modul (Excel) gibt es noch keinen Bedarf fuer die geplante Plugin-Struktur.
- **Handbook unveraendert lassen und Entscheidungen nur im ADR festhalten:** verworfen. Mehrere der aufgedeckten Luecken (Sicherheitsstufen-Tabelle, Roadmap-Text, Governance-Doku) sind Handbook-Inhalt, kein Code - ein ADR allein wuerde das Handbook veraltet/widerspruechlich zum tatsaechlichen Vorgehen zuruecklassen.

## Konsequenzen
- Naechster Schritt vor Excel-Code: technischer Vorschlag zur Freigabe durch PO (Bibliothek, Commands, Registry-Integration) - noch nicht Teil dieses ADR.
- `docs/AI_START.md` und `docs/PROJECT_STATE.md` verweisen ab jetzt auf `JARVIS_MASTER_HANDBOOK_v3_3.docx` statt `v3_2.docx`.
- Kuenftige Handbook-Aenderungen folgen der in Kap. 2 (v3.3) neu dokumentierten Regel: nur zwischen zwei Jarvis-Versionen.
