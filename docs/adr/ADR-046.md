# ADR-046: „Jarvis Command Center" — lokales read-only Dashboard mit echten Daten

## Problem/Kontext

Welle 4.3 des Offensiv-Fahrplans (PO-genehmigt 09.07.2026, Start 10.07.):
eine Status-Seite, die Jarvis' realen Betrieb sichtbar macht — Runtime-Status,
offene Einträge, Gedächtnis-Stand, Delegations-Kosten, heutige Aktivität.
Anspruch (Abgrenzung zu inszenierten Schaufenster-UIs): **jede Zahl echt**,
aus einer realen Quelle. Vereint drei Prinzipien: Beobachtbarkeit
(n8n-Lehre aus dem Architekturvergleich), sichtbares Gedächtnis (PIS) und
Schaufenster-Material (Welle 4).

## Entscheidung

1. **Eigener, strikt lesender Prozess** (`dashboard.py`), NICHT in der
   Runtime: null Eingriff in die stabile v1.0-Runtime, kein Lock-Erwerb,
   kein Schreiben. Liest die Dateien, die die Runtime atomar schreibt
   (`core/fileio.py`): `memory_data/`-JSONs, `logs/`-Textdateien,
   PROJECT_STATE-Kopf.
2. **Lock-Datei als Laufzeit-Signal:** fehlt → aus; unlesbar
   (msvcrt-Sperre der aktiven Instanz, ADR-026) → **an**; lesbar →
   PID-Lebendprüfung. Der Sperr-Effekt, der anderswo ein Hindernis ist,
   ist hier der Beweis.
3. **Nur `127.0.0.1`**, stdlib-only (`http.server`): keine neue Abhängigkeit,
   keine Netzwerk-Fläche, keine Auth nötig — die Seite existiert nur auf
   POs Rechner. Port konfigurierbar (`dashboard_port`, Default 8765).
4. **Datensammlung getrennt von Darstellung:** `core/dashboard_data.py`
   (pure, pfad-injizierbar, ohne HTTP testbar) liefert `/api/status`-JSON;
   die HTML-Seite (dunkles Cockpit, inline CSS/JS, kein CDN) pollt alle 5 s.
5. **Privacy wie überall:** Es erscheinen Kennzahlen und Eintrags-/Fakten-
   Texte, die der PO selbst gespeichert hat — auf einer localhost-Seite
   seines eigenen Rechners. Log-Auswertung erhebt nur Zählwerte, nie
   Inhalte (Transkript-Grundsatz bleibt).

## Begründung

Kleinste ehrliche Form eines Command Centers: vorhandene Datenquellen
sichtbar machen statt neue Telemetrie erfinden. Der getrennte Prozess hält
das Risiko exakt bei null — die Runtime weiß nicht einmal, dass es das
Dashboard gibt.

## Alternativen erwogen

- **HTTP-Server als Thread in der Runtime:** verworfen — live-Zugriff auf
  interne Zustände wäre schöner (laufende Delegation), aber jede neue
  Fläche in der Runtime gefährdet die frisch getaggte v1.0; Logs decken
  die Fälle ausreichend.
- **Flask/FastAPI:** verworfen — eine Abhängigkeit für eine Seite mit zwei
  Routen ist nicht zu rechtfertigen (stdlib genügt).
- **Elektron-/Desktop-App:** verworfen — Browser auf localhost tut es.

## Konsequenzen

- Kennzahlen sind so aktuell wie die Dateien (Poll 5 s) — „laufende
  Delegation live" gibt es in v1 nicht, nur abgeschlossene aus den Logs.
- Zwei Prozesse können dieselben Dateien lesen/schreiben — unkritisch, da
  die Runtime atomar schreibt und das Dashboard nur liest.
- Der Server läuft, bis sein Fenster geschlossen wird (bewusst sichtbar,
  kein Hintergrund-Dienst).

## Falsifizierbarkeit

Zu revidieren, falls (a) das Dashboard reale Bedarfe zeigt, die nur die
Runtime beantworten kann (dann Runtime-API als eigenes, freizugebendes
Paket), (b) das 5-s-Polling die Maschine spürbar belastet, oder (c) der
Kill-Kriterium-Standard greift (2 Wochen unbenutzt → raus).

## Bewusst NICHT in diesem Schritt

Runtime-interne Live-Zustände (laufende Delegation, Queue-Tiefe) ·
Netzwerk-Zugriff/Auth · Schreib-Aktionen aus dem Dashboard (reines
Anzeigen, keine Fernbedienung) · Historien-Charts · Autostart des Servers.
