# ADR-005: Piper TTS für Sprachausgabe (v0.3 Definition of Done)

## Problem
Die v0.3 Definition of Done (Handbook Kap. 28) verlangt "Piper TTS
ersetzt pyttsx3". Tatsächlich enthielt die v0.2-Foundation gar kein
pyttsx3 mehr - `core/speech.py` war bereits reine Konsolen-Ein-/
Ausgabe (bewusster Platzhalter, siehe Moduldoc). Es musste also nicht
ersetzt, sondern erstmals eine echte Sprachausgabe ergänzt werden.

## Entscheidung
`SpeechEngine.say()` gibt Text weiterhin immer auf der Konsole aus
(Transkript/Log bleibt erhalten) und spricht ihn zusätzlich über
Piper (lokal, offline, kein Cloud-Dienst), wenn:
1. `piper-tts` installiert ist (`pip install piper-tts`),
2. `tts_enabled: true` in config.json gesetzt ist,
3. das konfigurierte Sprachmodell (`tts_model_path`) tatsächlich
   existiert, und
4. Jarvis unter Windows läuft (Wiedergabe über `winsound`, stdlib).

Fehlt eine dieser Voraussetzungen, fällt Jarvis automatisch und ohne
Absturz auf reine Konsolenausgabe zurück - `tts_enabled` ist deshalb
per Default `false`, bis das Modell manuell heruntergeladen wurde
(README, Abschnitt "Piper TTS einrichten").

`listen()` (Spracheingabe) bleibt bewusst Konsole - Mikrofon/Wake-Word
ist ein eigenes Feature unter "Next" (Kap. 27), nicht Teil dieser ADR.

## Begründung
- Regel 6 (keine Architecture Astronautics): kein automatischer
  Hintergrund-Download des Modells beim Programmstart (das wäre
  "Magie" im Sinne von Design Principle 6 - PO muss jede
  Komponente verstehen können). Stattdessen ein expliziter, sichtbarer
  Einmal-Befehl in der README.
- `winsound` (Windows-only, stdlib) statt einer plattformübergreifenden
  Audio-Bibliothek: Jarvis existiert laut Produktvision (Kap. 26)
  ausschließlich für PO auf seinem Windows-Rechner - keine
  zusätzliche Abhängigkeit für Portabilität, die niemand braucht
  (YAGNI).
- Graceful Degradation statt harter Abhängigkeit: Piper/Modell fehlen
  in jeder Test-/CI-Umgebung (kein Windows, kein Modell-Download) -
  Jarvis muss trotzdem lauffähig und testbar bleiben.

## Konsequenzen
- Sprachausgabe funktioniert nur unter Windows. Unter Linux/Mac bleibt
  es bei Konsolenausgabe (kein Fehler, nur kein Ton).
- Ein zusätzliches, separates Downloadartefakt (Sprachmodell, ca.
  60 MB) muss einmalig manuell besorgt werden.

## Falsifizierbarkeit
Gilt als unzureichend, wenn der manuelle Setup-Schritt in der Praxis
zu unbequem ist (z. B. PO das Modell nie herunterlädt) - dann
Review: evtl. ein `python -m piper --model ... --download-dir voices`
Onboarding-Skript statt reiner README-Anleitung ergänzen (v0.4).
