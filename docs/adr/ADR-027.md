# ADR-027: Runtime v2 - TelegramChannel als erster echter Runtime-Kanal

## Problem
Runtime v1 (ADR-025) instanziiert den Core-Stack einmalig und verarbeitet Nachrichten über `queue.Queue` + einen Worker-Thread - bewiesen durch den einzigen vorhandenen Kanal, `ConsoleDummyChannel`. Dieser Kanal blockiert auf `input()` und ist damit für einen unbeaufsichtigt laufenden Prozess ungeeignet: eine Bewertung nach Freigabe von Runtime v1 (siehe `docs/logbook.md`, 02.07.2026) ergab, dass ein automatisch gestarteter Jarvis zwar durch den Single-Instance-Schutz (ADR-026) sicher als Einzelinstanz liefe, aber ohne einen Kanal, der ohne angehängte Konsole nutzbar ist, faktisch unerreichbar wäre. Das ist die eigentliche, bisher fehlende Voraussetzung für einen künftigen Jarvis-Eigenstart-Baustein - **nicht** der Eigenstart-Mechanismus selbst, der bewusst ein eigener, späterer Schritt bleibt.

Ein Architekturvorschlag wurde erarbeitet und als Richtung freigegeben (siehe `docs/logbook.md`): Runtime v2 fügt genau einen neuen, echten Kanal hinzu - Telegram, weil dessen Sicherheits- und Transportlogik bereits vollständig implementiert und produktiv erprobt ist (`telegram_main.py`, ADR-018).

## Entscheidung

### 1. Welches Problem löst Runtime v2?
Runtime v2 macht die Runtime zum ersten Mal tatsächlich ohne angehängte Konsole nutzbar. Das Grundgerüst selbst (Queue, Worker-Thread, sauberer Shutdown, Single-Instance-Schutz) ist bereits vorhanden und bleibt unverändert - Runtime v2 fügt **ausschließlich** die fehlende Erreichbarkeit hinzu, indem sie einen zweiten, echten Kanal einbindet.

### 2. Warum ist Telegram bereits Bestandteil von Runtime v2 (nicht ein separater, späterer Schritt)?
Ein "Framework zuerst, Kanal später"-Schnitt wurde geprüft und verworfen (siehe Alternativen): Die Mehrkanal-Fähigkeit der Runtime ist durch `submit(text, reply_callback)` und die bereits in Runtime v1 getesteten nebenläufigen `submit()`-Aufrufe (`tests/test_jarvis_runtime.py::test_concurrent_submits_from_multiple_threads_not_lost`) bereits bewiesen. Ein isolierter "Framework"-Schritt ohne einen echten zweiten Kanal hätte nichts Neues zu validieren und liefe Gefahr, Struktur (z. B. ein Channel-Interface) auf Vorrat zu bauen, bevor ein echter zweiter Verwender ihre tatsächlichen Anforderungen zeigt - genau das widerspricht Regel 6 (keine Architecture Astronautics) und dem in ADR-024 selbst festgelegten Prinzip ("Channel-Interface erst beim zweiten echten Kanal"). Telegram liefert als einziger bereits vorhandener, produktiv nutzbarer, konsolenfreier Kanal den einzigen echten Erkenntnisgewinn, den ein zweiter Kanal bringen kann.

### 3. Warum wird dafür kein Channel-Interface eingeführt?
`submit(text, reply_callback)` bleibt die vollständige Kopplungsstelle zwischen Runtime und Kanälen - `JarvisRuntime` muss auch mit `TelegramChannel` nicht wissen, dass es sich um Telegram handelt. `ConsoleDummyChannel` und `TelegramChannel` haben grundverschiedene Lebenszyklus-Modelle (synchrone blockierende Konsolen-Schleife vs. `python-telegram-bot`s eigener, asynchroner Event-Loop über `run_polling()`) - ein gemeinsames formales Interface würde entweder nur eine leere Hülle sein (kein gemeinsames Verhalten außer `submit()`, das ohnehin schon auf `JarvisRuntime` liegt) oder kanal-spezifische Eigenheiten künstlich glätten. Ein Interface liefert hier nur Dokumentations-, keinen Verhaltenswert - das ist eine bewusste Abweichung von ADR-024s eigener Erwartung ("zweiter Kanal = Anlass fürs Interface"), die hiermit explizit begründet wird, statt automatisch übernommen zu werden.

### 4. Neue/geänderte Dateien (spätere Umsetzung)
- **Neu: `telegram_channel.py`** (Top-Level, neben `main.py`/`telegram_main.py`/`jarvis_runtime.py` - kein neues Package, gleicher Präzedenzfall wie die flache Struktur seit ADR-013/ADR-025). Enthält `TelegramChannel` - die einzige Stelle im Runtime-Umfeld, die `python-telegram-bot` importiert und die Asyncio-Brücke (siehe Punkt 6) implementiert. Telegram-spezifischer Code bleibt damit vollständig von `jarvis_runtime.py` getrennt und in dieser einen Datei gekapselt.
- **Neu: `tests/test_telegram_channel.py`**.
- **Geändert: `jarvis_runtime.py`**
  - `JarvisRuntime.submit()`/`_process()` erhalten einen optionalen, generischen Parameter (ein Filter-Callable `steps -> (erlaubte_steps, Ablehnungsgrund)`), den ein Kanal beim Aufruf übergeben kann. `ConsoleDummyChannel` übergibt ihn weiterhin nicht (Default `None`) - Verhalten für den bestehenden Kanal bleibt exakt wie in Runtime v1, vollständig rückwärtskompatibel. `JarvisRuntime` kennt den Inhalt des Filters nicht (kein Telegram-Wissen im Core der Runtime) - nur die Erweiterungsstelle selbst ist neu.
  - `main()` startet künftig zusätzlich zu `ConsoleDummyChannel` (unverändert im Hauptthread) optional `TelegramChannel` in einem eigenen Thread, falls die bekannten Umgebungsvariablen (`JARVIS_TELEGRAM_BOT_TOKEN`, `JARVIS_TELEGRAM_ALLOWED_CHAT_ID`) gesetzt sind - fehlen sie, verhält sich `main()` exakt wie in Runtime v1 (nur Konsole).
- **Unverändert:** `docs/CHANGELOG.md`, `docs/logbook.md`, `README.md`, `docs/PROJECT_STATE.md` werden erst bei der eigentlichen Umsetzung (nach separater Freigabe) aktualisiert - nicht durch diese ADR.

### 5. Was bleibt ausdrücklich unverändert?
- `core/*` (`ai.py`, `planner.py`, `config.py`, `tool_manager.py`, `models.py`, `speech.py`) - keine Änderung.
- `executor/executor.py` - keine Änderung.
- `commands/*` - keine Änderung.
- `memory/*` (`store.py`, `long_term.py`) - keine Änderung.
- `core/single_instance.py` - keine Änderung; der Lock schützt automatisch weiter, unabhängig vom Kanal.
- `telegram_main.py` - keine Änderung, bleibt vollständig eigenständig lauffähig (Koexistenz, wie in ADR-024 festgelegt und seither wiederholt bestätigt).
- `main.py` - keine Änderung.
- `_RuntimeSpeech` in `jarvis_runtime.py` - keine Änderung; da `TelegramChannel` ausschließlich über `runtime.submit()` in den bereits bestehenden, geteilten `Executor` (mit `_RuntimeSpeech`) läuft, gilt der fail-closed-Schutz für Sicherheitsstufe 2/3 automatisch auch für Telegram-Nachrichten - **ohne** dass `telegram_channel.py` einen eigenen Speech-Adapter braucht (Unterschied zu `telegram_main.py`, das einen eigenen `Executor` und deshalb eine eigene `TelegramSpeech`-Instanz benötigt).

### 6. Asyncio-Brücke zwischen Runtime-Worker und `python-telegram-bot` (explizit dokumentiert)
`python-telegram-bot` (Version 20+, bereits Projektabhängigkeit seit ADR-018) ist strukturell asynchron - `Application.run_polling()` betreibt einen eigenen `asyncio`-Event-Loop, Nachrichten-Handler sind Koroutinen. Die Runtime selbst bleibt bewusst synchron/Thread-basiert (ADR-024, KISS, Product-Owner-Entscheidung gegen `asyncio` für die Runtime) - das ändert sich durch diese ADR **nicht**. `TelegramChannel` bringt seinen eigenen Event-Loop mit, weil das eine Eigenschaft der Bibliothek ist, nicht weil die Runtime asynchron würde.

Die Brücke besteht aus zwei Richtungen:
- **Eingehend (Telegram → Runtime):** Der asynchrone Nachrichten-Handler von `python-telegram-bot` ruft nach der Whitelist-Prüfung (Punkt 7) synchron `runtime.submit(text, reply_callback, plan_filter=...)` auf - `submit()` selbst ist bereits thread-safe (`queue.Queue`), ein Aufruf aus einer Koroutine heraus ist unproblematisch, solange er nicht blockiert (er blockiert nicht - `submit()` legt nur ein Tupel in die Queue).
- **Ausgehend (Runtime → Telegram):** Der `reply_callback` wird vom Runtime-Worker-Thread aufgerufen - einem anderen Thread als dem, der den Event-Loop von `python-telegram-bot` besitzt. Das eigentliche Senden (`context.bot.send_message(...)`) ist eine Koroutine und darf nicht direkt aus einem fremden Thread aufgerufen werden. `TelegramChannel` löst das über `asyncio.run_coroutine_threadsafe(coro, loop)` - die einzige Stelle im gesamten Projekt, an der zwei nebenläufige Ausführungsmodelle (Thread + Event-Loop) bewusst verbunden werden. Der Loop wird beim Start von `TelegramChannel` erfasst und für die Laufzeit gehalten.

Diese Brücke ist die technisch komplexeste Einzelkomponente, die dieses Projekt bisher bekommen würde (siehe Risiken, Punkt 8. weiter unten sowie Begründung).

### 7. Wiederverwendung der Sicherheitslogik statt Duplizierung
`telegram_channel.py` **importiert** die bereits bestehenden, reinen Funktionen/Konstanten aus `telegram_main.py` unverändert: `ALLOWED_INTENTS`, `filter_plan`, `rejection_reason`, `is_authorized`. Diese Funktionen haben keine Abhängigkeit von PTB-Laufzeitobjekten (sie arbeiten auf `Plan`/`str`), ein Import zieht keine zusätzliche Kopplung.

Diese Prüfung muss **vor** dem eigentlichen `submit()`-Aufruf stattfinden, nicht danach: Anders als `telegram_main.py`s `JarvisBridge`, das den Plan selbst berechnet und bei Ablehnung den `Executor` gar nicht erst aufruft, kennt `JarvisRuntime._process()` keinerlei Kanal- oder Whitelist-Konzept - jede über `submit()` eingereichte Nachricht würde ansonsten identisch zu einer Konsolen-Eingabe behandelt und **jeden** Intent erreichen (nicht nur Sicherheitsstufe-2/3-Commands, die `_RuntimeSpeech` ohnehin fail-closed blockiert, sondern auch bestätigungsfreie, aber laut `ALLOWED_INTENTS` nicht für Telegram vorgesehene Intents wie `read_excel`, `analyze_report`, `analyze_pc`). Deshalb bekommt `submit()`/`_process()` den in Punkt 4 beschriebenen optionalen Filter-Parameter: `TelegramChannel` lässt den Planner (`runtime.planner`, bereits öffentlich, unverändert) einmalig planen, wendet `filter_plan()` darauf an und übergibt das Ergebnis als Filter an `submit()` - wird der gesamte Plan abgelehnt, antwortet `TelegramChannel` direkt mit dem Ablehnungsgrund, ohne dass `_process()` den `Executor` je aufruft.

Warum keine Duplizierung (wie noch bei `_RuntimeSpeech`/`TelegramSpeech`, ADR-025): Der damalige Grund fürs Duplizieren war ausschließlich, `python-telegram-bot` nicht als Abhängigkeit in die Runtime zu ziehen. Dieser Grund entfällt hier von vornherein, da `telegram_channel.py` PTB ohnehin zwingend braucht (es kapselt `Application`). Ein Duplizieren der Whitelist-Logik hätte hier nur das Risiko geschaffen, dass beide Implementierungen bei einer künftigen Erweiterung der Whitelist auseinanderlaufen - ohne kompensierenden Vorteil. Zusätzlich bleibt die fail-closed-Sperre für Sicherheitsstufe-2/3-Commands (`_RuntimeSpeech`, Punkt 5) als zweite, unabhängige Schutzschicht bestehen - selbst falls die Whitelist-Prüfung fehlerhaft wäre, würde ein bestätigungspflichtiger Command trotzdem sicher abgelehnt (Design Principle 4, Safety First - mehrschichtiger statt einstufiger Schutz).

### 8. Bewusst NICHT enthalten
Windows-Autostart (HKCU Run-Key, `enable_/disable_jarvis_autostart`-Intents, Sicherheitsstufe 2) - "Jarvis-Eigenstart" im engeren Sinn bleibt ein eigener, späterer Vorschlag nach Freigabe/Umsetzung dieser ADR. Kein Tray, kein eigenes UI, kein Wake-Word. Kein Runtime-v3-Scope. Kein formales Channel-Interface/Protocol (Punkt 3). Keine Erweiterung des Telegram-Sicherheitsmodells (z. B. mehrere Chat-IDs) - 1:1-Übernahme des bestehenden Modells. Keine Änderung an `telegram_main.py` selbst. Keine Persistenz/Wiederherstellung der Runtime-Queue über Neustarts hinweg.

## Begründung
- Telegram statt eines neuen, minimaleren Kanals: nutzt bereits vorhandenen, produktiv erprobten Code (Transport **und** Sicherheitslogik), liefert echten Mehrwert (Fernzugriff, passend zur langfristigen "immer erreichbar"-Vision) statt nur eine weitere Beweis-Kanal-Attrappe wie `ConsoleDummyChannel`.
- Erweiterung von `submit()` um einen optionalen Filter-Parameter statt eines Channel-Interfaces oder direkter Channel-Zugriffe auf `runtime.planner`/`runtime.memory`: hält "Runtime kommuniziert ausschließlich über `submit()`" wörtlich ein, vermeidet eine zweite, redundante Planungs-Anfrage an die KI pro Nachricht (die bei einem naiven "Kanal prüft selbst, dann ruft submit() blank auf"-Ansatz nötig wäre) und hält `JarvisRuntime` weiterhin vollständig kanal-agnostisch (der Filter ist ein generisches Callable, kein Telegram-Wissen).
- Wiederverwendung statt Duplizierung der Whitelist-Logik: Design Principle 6 (Keine Magie) und die im Architekturvorschlag explizit benannte Divergenz-Gefahr zweier unabhängiger Sicherheitsimplementierungen - der frühere Duplizierungsgrund (PTB-Abhängigkeit vermeiden) entfällt hier strukturell.
- Zwei unabhängige Schutzschichten (Whitelist-Filter **und** fail-closed `_RuntimeSpeech`) statt einer: Design Principle 4 (Safety First) - kein Single Point of Failure für die Telegram-Sicherheit.
- Kein Channel-Interface: Regel 6 (keine Architecture Astronautics) - mit zwei grundverschiedenen Kanal-Lebenszyklen liefert eine gemeinsame Abstraktion aktuell nur Dokumentations-, keinen Verhaltenswert.

## Alternativen erwogen
- **Runtime v2 als reines "Mehrkanal-Framework" ohne Telegram, Telegram als separater, späterer Schritt:** verworfen (Punkt 2) - die Mehrkanal-Fähigkeit ist durch Runtime v1 bereits bewiesen, ein telegramloser Zwischenschritt hätte nichts Neues zu validieren und liefe Gefahr, Struktur auf Vorrat zu bauen.
- **Ein noch minimalerer neuer Kanal** (lokaler Named-Pipe-/Socket-Kanal statt Telegram): verworfen als Hauptweg - löst nicht das eigentliche Bedürfnis (Fernzugriff), nutzt keinen vorhandenen Code, liefert weniger Mehrwert für vergleichbaren Aufwand. Bliebe eine Option, falls die Asyncio-Brücke (Punkt 6) als zu hohes Risiko eingestuft wird - das wurde dem Product Owner vor dieser ADR vorgelegt und nicht gewählt.
- **`TelegramChannel` prüft die Whitelist selbst über direkten Zugriff auf `runtime.planner`/`runtime.memory`, ruft danach unverändertes `submit(text, reply_callback)` auf:** verworfen - verletzt "Runtime kommuniziert ausschließlich über `submit()`" im Kern, verursacht eine doppelte (kostenpflichtige) KI-Planungs-Anfrage pro erlaubter Nachricht, vergrößert die Kopplungsfläche zwischen Kanal und Runtime unnötig.
- **Whitelist-Logik in `telegram_channel.py` duplizieren statt zu importieren:** verworfen (Punkt 7) - der ursprüngliche Duplizierungsgrund (PTB-Abhängigkeit vermeiden) entfällt hier, Duplizierung hätte nur Divergenz-Risiko ohne Nutzen geschaffen.
- **Formales `Channel`-Interface/Protocol bereits jetzt einführen:** verworfen (Punkt 3) - verfrühte Abstraktion, kein Verhaltenswert bei nur zwei, strukturell verschiedenen Kanälen.
- **Telegram-Handling direkt in `jarvis_runtime.py` statt in eigener Datei:** verworfen - widerspricht der expliziten Vorgabe, Telegram-spezifischen Code vollständig von der Runtime getrennt zu halten, und würde `python-telegram-bot` zu einer Pflichtabhängigkeit für jeden `jarvis_runtime.py`-Import machen (aktuell optional, nur bei tatsächlicher Nutzung von `telegram_channel.py` relevant).

## Risiken
- Die Asyncio-Thread-Brücke (Punkt 6) ist die komplexeste Einzelkomponente, die dieses Projekt bisher bekäme - höheres Risiko für subtile Fehler (z. B. Deadlocks, falls der Runtime-Worker blockiert, während der Event-Loop auf eine Antwort wartet) als bei den bisher rein synchronen Komponenten. Muss bei der Umsetzung mit eigenen, gezielten Tests abgesichert werden.
- **Telegram-API-Konflikt bei doppeltem Long-Polling:** Telegram erlaubt pro Bot-Token nur eine aktive `getUpdates`-Verbindung gleichzeitig. Werden `telegram_main.py` und `jarvis_runtime.py` (mit `TelegramChannel`) versehentlich gleichzeitig mit demselben Bot-Token gestartet, meldet Telegrams Server einen Konflikt. Der Single-Instance-Schutz (ADR-026) verhindert dies bereits indirekt, solange beide Prozesse das gleiche `memory_dir` verwenden (Standardfall) - bleibt aber ein eigenständiges Risiko, das in der README bei der Umsetzung explizit dokumentiert werden sollte.
- Zwei parallele Wege, Telegram zu nutzen (`telegram_main.py` eigenständig vs. `TelegramChannel` in der Runtime) - kein Architekturbruch (Koexistenz ist gewollt), aber Klarheits-/Doku-Risiko, welcher Weg wann genutzt werden soll.
- `submit()`/`_process()` bekommen mit dem optionalen Filter-Parameter erstmals eine Erweiterungsstelle - muss sorgfältig so gebaut werden, dass `ConsoleDummyChannel` (und jeder künftige einfache Kanal ohne Whitelist-Bedarf) exakt unverändert weiterläuft, ohne den Filter übergeben zu müssen.
- `python-telegram-bot` wird für `telegram_channel.py` zur Pflichtabhängigkeit - muss beim fehlenden Paket sauber und optional degradieren (Runtime mit nur `ConsoleDummyChannel` muss weiterhin ohne PTB-Installation funktionieren).

## Konsequenzen
- Neue Datei `telegram_channel.py`, neue Testdatei `tests/test_telegram_channel.py` (bei der eigentlichen Umsetzung, nach separater Freigabe).
- `jarvis_runtime.py` wird ein zweites Mal geändert (nach der Single-Instance-Integration) - `submit()`/`_process()` um einen optionalen Filter-Parameter erweitert, `main()` startet optional zusätzlich `TelegramChannel`. Rückwärtskompatibel zu Runtime v1.
- **Keine** Änderung an `core/*`, `executor/executor.py`, `commands/*`, `memory/*`, `core/single_instance.py`, `telegram_main.py`, `main.py`.
- Jarvis-Eigenstart (Windows-Autostart) bleibt weiterhin ein eigener, späterer Vorschlag - durch diese ADR nicht vorweggenommen.
- Ein dritter echter Kanal (Tray, UI, Wake-Word) bleibt der natürliche Anlass, die Entscheidung gegen ein Channel-Interface (Punkt 3) erneut zu prüfen.
- Diese ADR ist eine reine Architekturentscheidung - keine Implementierung. Umsetzung erfolgt erst nach separater Product-Owner-Freigabe.
