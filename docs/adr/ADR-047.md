# ADR-047: BrowserChannel — lokale Runtime-API für das Jarvis-UI

## Problem/Kontext

Das Command Center (ADR-046) ist eine Anzeigetafel: strikt lesend, von der
Runtime entkoppelt. Der PO-Auftrag (10.07.2026, Freigabe mit „2× Ja":
Vollzugriff + Orb) macht daraus **Jarvis' Gesicht**: ein Browser-UI mit
Chat-Eingabe und einem Orb, der den Live-Zustand zeigt (bereit / arbeitet /
wartet auf Bestätigung). Dafür braucht die Runtime erstmals eine
**ansprechbare lokale Schnittstelle** — genau der Revisionsfall, den
ADR-046 („Falsifizierbarkeit") vorgesehen hat.

## Entscheidung

1. **Vierter Kanal `BrowserChannel`** (`browser_channel.py`) — exakt das
   bewährte Kanal-Muster (ADR-027/Telegram): eigener Server-Thread,
   `runtime.submit()` als einziger Weg hinein, `reply_callback` als Weg
   hinaus. Die Runtime kennt den Browser nicht.
2. **Transport stdlib-only:** `POST /message` (JSON `{"text": …}`) reicht
   eine Nachricht ein; `GET /events` ist ein **Server-Sent-Events-Strom**
   (Antworten, Bestätigungs-Rückfragen, Zustandswechsel als JSON-Events).
   Kein WebSocket-Paket, kein Framework — `http.server` genügt; SSE
   reconnectet im Browser von selbst.
3. **Vollzugriff (PO-Entscheidung):** kein `plan_filter` — der Browser auf
   127.0.0.1 ist so vertrauenswürdig wie die Konsole am selben Rechner.
   Stufe-2/3 bleiben durch den Executor-Dialog gedeckt: der Kanal hat ein
   eigenes `ConfirmationGate` (ADR-045-Muster) — die Rückfrage kommt als
   `confirm`-Event, die nächste gesendete Nachricht ist die Antwort und
   geht nie durch den Planner.
4. **Absicherung:** Bind ausschließlich `127.0.0.1`. Zusätzlich
   **Origin-Prüfung** gegen browserbasierte Angriffe (eine fremde Webseite
   im selben Browser darf die lokale API nicht aufrufen): erlaubt sind nur
   fehlender Origin (curl/Skripte) und `http://127.0.0.1:*` /
   `http://localhost:*`; alles andere wird mit 403 verworfen, ohne die
   Nachricht zu verarbeiten.
5. **Zustands-Events für den Orb:** der Kanal meldet `arbeitet` beim
   Einreichen, `bereit` nach der Antwort, `wartet` bei offener
   Stufe-2/3-Rückfrage. Feinere Zustände („hört zu"/„spricht" vom
   Wake-Word) folgen in einer späteren Scheibe über denselben Event-Strom.
6. **Aus per Default:** `ui_enabled` ist im Code `false` (neue
   Kontroll-Fläche ⇒ bewusst einschalten, gleiches Prinzip wie
   `wake_word_enabled`/ADR-037); Port `ui_port` (Default 8766). Beim PO an.

## Begründung

Kleinste tragfähige Form eines lebendigen UIs: ein Kanal wie jeder andere,
dieselbe Pipeline, dieselben Sicherheitsstufen — nur der Transport ist neu.
SSE statt WebSocket spart die einzige nötige Abhängigkeit und reicht für
Chat-Frequenzen vollkommen.

## Alternativen erwogen

- **WebSocket (`websockets`-Paket):** verworfen — bidirektional brauchen
  wir nicht (Eingaben gehen über POST), und SSE ist stdlib-machbar.
- **UI-Server im Dashboard-Prozess mit Datei-Briefkasten zur Runtime:**
  verworfen — Polling-Latenz, zwei Wahrheiten, hässliches Locking.
- **Vollzugriff ohne Origin-Prüfung:** verworfen — „localhost-only" schützt
  nicht vor dem eigenen Browser als verwirrtem Stellvertreter.

## Konsequenzen

- Die Runtime bekommt erstmals eine ansprechbare Fläche — bewusst als
  optionaler Kanal (Default aus), Muster identisch zu Telegram, kein
  neuer Sicherheitspfad.
- Das UI (Scheibe 2) kann degradieren: Runtime aus ⇒ kein Event-Strom ⇒
  die Seite zeigt den read-only-Stand von ADR-046 weiter.
- Mehrere offene Browser-Tabs teilen sich den Strom (Broadcast) — die
  Antwort erscheint in allen; akzeptiert (Ein-Personen-System).

## Falsifizierbarkeit

Zu revidieren, falls (a) SSE sich im Alltag als instabil erweist
(Reconnect-Lücken ⇒ WebSocket), (b) die Origin-Prüfung legitime Nutzung
blockiert, oder (c) der Vollzugriff im Nutzungslauf zu Fehlauslösungen
führt (⇒ Whitelist wie Telegram).

## Bewusst NICHT in diesem Schritt

Das UI selbst (Scheibe 2) · „hört zu"/„spricht"-Events vom Wake-Word ·
Auth/Netzwerkzugriff · Verlaufs-Persistenz im Browser · mehrere
gleichzeitige Bestätigungen.
