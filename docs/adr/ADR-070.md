# ADR-070: Skill-Bibliothek — von Einweg-Bau zu wachsendem Können (Plan A1)

## Problem / Kontext
Die Königsdisziplin (ADR-067/069) lässt Jarvis Werkzeuge bauen — aber jeder Bau
blieb ein **isoliertes** Projekt. Das Deep Research (13.07., `docs/proposals/
deep-research-2026-07-13.md`) zeigt das stärkste Muster der selbst-verbessernden
Agenten 2026: eine **Skill-Bibliothek** (Voyager/SAGE), die gelöste Aufgaben als
benannte, wiederverwendbare Fähigkeiten sammelt und VOR dem Neubauen prüft — so
wächst Können *kompound* statt punktuell. Anthropic „Agent Skills" ist inzwischen
ein Standard (Claude Code = Jarvis' Bau-Arm kennt ihn).

## Entscheidung
**Stufe A1 (dieser ADR, sicher):** Jarvis registriert erfolgreich Gebautes als
Skill und macht die Bibliothek sicht- und nutzbar für die Vorschlags-Logik.
- `memory/skills.py::SkillLibrary` (file-backed, `memory_dir/skills.json`):
  `add` (Dedup über Namen), `all`/`names`/`get`, `find_similar` (Embedding,
  für „schon gebaut?"). Fail-safe.
- `commands/delegate.build_project` registriert bei Erfolg `{name, beschreibung,
  pfad, datum}` — fail-safe, stört den Bau nie.
- `commands/skills.list_skills` („was hast du schon gebaut?") listet die Bibliothek
  (nur lesend, in der Kern-Whitelist).
- `core/build_suggestion.suggest_build(existing_skills=…)`: der proaktive Bau-
  Vorschlag bekommt die bereits gebauten Namen und schlägt **nichts erneut** vor
  („prüfen vor dem Bau" auf Vorschlags-Ebene). ADR-069 deckt den Fall auf
  Ausführungs-Ebene ab (gleicher Name → Gerüst wird aufgegriffen).

**Stufe A2 (später, GATED):** einen gebauten Skill auf Zuruf AUSFÜHREN. Das ist
Code-Ausführung außerhalb des Käfigs = S4b-Territorium (Werkzeug-Parität, PO-gated)
— bewusst NACH Stufe 2, nicht hier.

## Sicherheit
- A1 ist reiner Metadaten-/Lookup-Layer — **keine Ausführung**, kein neuer
  Blast-Radius. Registrierung + Bibliothek-Lesen sind fail-safe.
- Der Bau selbst bleibt gated (ADR-067/069): requires_confirmation, Sandbox,
  Selbstprüfung, Marker-Wächter, Selbst-Repo gesperrt.
- Optionaler `SKILL.md`-Ausbau (Anthropic-Format) ist Interop-Vorbereitung, kein
  Verhalten hier.

## Falsifizierbarkeit
Unzureichend, wenn die Bibliothek nervt (Doppelvorschläge trotz Dedup) oder falsch
matcht (find_similar zu grob/fein) → Schwelle/Prompt schärfen. Evidenz:
Registrierung im Bau-Test verifiziert; `list_skills` am echten Kern (3/3); Dedup
im Vorschlag über die Namensliste.

## Status
Accepted (PO-Go „ganzen sicheren Batch"). A1 gebaut; A2 gated/später. Siehe
[[angestellten-vision]], [[jarvis-projektentwickler-plan]], ADR-067/069.
