# ADR-051: Merk-Angebot - proaktiver Gedaechtnis-Vorschlag mit verfallendem Angebot

## Problem/Kontext

PO-Vision (10.07.2026): "Mir ist wichtig, dass er auf Dauer meine
Gewohnheiten lernt. Wie ein Mensch es tun wuerde. Dann sollte er mich,
wenn er etwas vermutet, fragen." Heute speichert Jarvis dauerhafte Fakten
nur auf explizites "merk dir ..." (remember_fact, ADR-009: bewusst KEINE
automatische Fakten-Extraktion). Nebenbei erwaehnte Gewohnheiten/Vorlieben
("ich trinke meinen Kaffee uebrigens immer schwarz") gehen verloren, wenn
der Nutzer nicht an den Merk-Befehl denkt.

## Entscheidung

Stufe 1 des Gewohnheits-Lernens: das **Merk-Angebot**.

1. **Erkennung im Planner-Huckepack:** Der Intent-Planner (sieht ohnehin
   jede Nachricht) liefert ein optionales JSON-Feld `memory_suggestion`,
   wenn die Eingabe NEBENBEI einen dauerhaften persoenlichen Fakt enthaelt
   (Gewohnheit/Vorliebe/Beziehung). Kein zusaetzlicher API-Aufruf.
2. **Vorschlag statt Aktion:** Die Runtime haengt an die normale Antwort
   die Frage "Soll ich mir dauerhaft merken: <Fakt>? (ja/nein)". Es wird
   NIEMALS automatisch gespeichert (Governance-Invariante, PIS-Prinzip).
3. **Verfallendes Angebot:** Das Angebot schwebt genau bis zur naechsten
   Nachricht. Eindeutiges "ja" speichert (remember_fact, Kategorie
   gewohnheit), eindeutiges "nein" schreibt den Fakt auf die Nein-Liste
   (`memory_data/memory_declined.json`) - er wird nie wieder angeboten.
   JEDE andere Nachricht laesst das Angebot still verfallen und wird
   normal verarbeitet (ignorieren ist eine gueltige Antwort).
4. **Nerv-Schutz:** Kein Angebot fuer Fakten, die bereits im
   Langzeitgedaechtnis stehen (Teilstring-Heuristik in beide Richtungen)
   oder auf der Nein-Liste. Config-Schalter `memory_offers_enabled`,
   Default AUS (neues Verhalten, bewusst einschalten - Muster
   wake_word_enabled).

## Begruendung

- Kein Extra-Aufruf: Latenz und Kosten bleiben unveraendert; der Planner
  traegt die Erkennung als Nebenprodukt.
- Nicht-blockierend: Der Bestaetigungs-Gate-Weg (ADR-045) wuerde den
  Kanal auf eine Antwort festnageln - ein optionales Angebot darf
  ignoriert werden, sonst nervt es. Deshalb ein eigener, verfallender
  Mechanismus statt des Gates.
- Sprachtauglich: Das "ja" im Anschluss-Fenster genuegt; kein Tippen
  noetig.
- ADR-009 bleibt gewahrt: keine automatische Extraktion IN das
  Gedaechtnis - nur ein Vorschlag, der Nutzer entscheidet jedes Mal.

## Alternativen

- **Automatisch speichern + Undo:** verworfen - das Gedaechtnis ueber den
  Nutzer ist der sensibelste Datenbestand; hier gilt Zustimmung VOR der
  Tat (anders als bei Listen-Leeren, wo Undo genuegt).
- **Eigener LLM-Aufruf als Fakt-Detektor:** verworfen - verdoppelt
  Planner-Kosten/-Latenz fuer jede Nachricht.
- **Gate-basierte Rueckfrage (ADR-045):** verworfen - blockierend,
  Timeout-Zwang, fuer ein optionales Angebot unangemessen.

## Konsequenzen

- Plan-Modell traegt das Feld `memory_suggestion` (loest nie selbst eine
  Aktion aus - Trust Boundary wie gehabt, `confirmed` bleibt gestrippt).
- Die Runtime haelt genau EIN schwebendes Angebot (Worker ist seriell).
- Neue Datei `memory_data/memory_declined.json` (Nein-Liste, atomar).
- Fehlerkennungen des Planners kosten genau eine Frage und ein "nein" -
  danach herrscht fuer diesen Fakt fuer immer Ruhe.

## Falsifizierbarkeit

Gilt als gescheitert, wenn im Alltag die Fehlangebote ueberwiegen
(Nutzer lehnt haeufiger ab als an) oder das Angebot den Gespraechsfluss
stoert - dann wird `memory_offers_enabled` beim PO ausgeschaltet und die
Erkennungsregel im Planner-Prompt verschaerft oder das Feature entfernt
(Kill-Kriterium der Kampagne).

## Bewusst NICHT

- Kein Muster-Lernen ueber Zeit ("du fragst montags oft nach Reports") -
  das ist Stufe 2 mit eigener Datengrundlage und eigenem ADR.
- Keine automatische Speicherung, in keinem Fall.
- Keine Angebots-Warteschlange (ein Angebot zur Zeit; ein neues
  ueberschreibt kein altes, weil das alte beim naechsten Submit ohnehin
  verfaellt).

## Status

Accepted (PO-Freigabe 2026-07-10, "Go fuer den Plan").
